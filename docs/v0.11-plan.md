# v0.11 — the launch experience grows up

<!-- llm-parse: cycle=v0.11 status=open anchor=launch-phase-fidelity -->

## Context

v0.10 ("the fleet flies for real") shipped v0.10.0 → v0.10.7 and
closed with the v0.10.6/.7 perspective-tilt + launch-anchor pair —
which exposed that orbit-scale rendering still can't deliver two
reads the launch phase needs: **early trajectory shape** (the
gravity-turn arc relative to the ground) and **orientation cues at
human scale**. v0.11 turns from *flying the fleet* to *growing up
the launch experience*: the headline is **launch-phase fidelity** —
a chase-cam scene at human scale that subsumes v0.10.7's
launch-anchor rendering while keeping its sim primitives load-bearing.

Workflow framing:

> Watch the rocket sit on the pad with the LUT beside it; light the
> engine and see the tower fall away; ride the chase-cam through the
> gravity turn with the horizon curving below; coast through the
> Karman line and watch the auto-release hand you back to the orbit
> view with an actual orbit-shape read once one exists.

That makes the cycle's spine **chase-cam scene pipeline →
launch-site detail (LUT + all-vessels-in-SOI) → stage-aware
launch sprites → polish/SurfaceColor sweep**. v0.10.7 keeps doing
its job on the orbit screen (anchored ascent-profile read); v0.11
adds the *human-scale* read that orbit-scale can't produce.

**Feature set locked** (2026-05-24) — v0.11 is launch-view-only;
mission scripting stays deferred to its own future cycle (same
posture as v0.10). The slice contract below is committed; subsequent
fold-ins need explicit approval, same as v0.10. Slice 1 is fully
designed (this doc); Slices 2–4 are at design-pass depth with the
headline observations from the ADR-0002 grill banked — each will
get its own slice-open grill before implementation, per v0.10
convention.

## Locked decisions

<!-- llm-parse: decisions=locked -->

The architectural decision lives in
[`docs/adr/0002-launch-view-as-distinct-viewmode.md`](adr/0002-launch-view-as-distinct-viewmode.md) —
ViewLaunch is a distinct ViewMode subsuming v0.10.7's anchor
*rendering* while keeping `sim.LaunchAnchorPhi` and
`sim.LaunchMissionFloorM` as reusable sim primitives. ViewTilted
keeps the anchor as its yaw rule (unchanged at orbit scale);
ViewLaunch is the new chase-cam read.

Slice 1 design decisions resolved 2026-05-24 (this doc's grill pass):

- **Session sentinel is explicit `World.LaunchSessionActive bool`.**
  `PrevViewMode` (a `ViewMode`) can't carry "no active session" as a
  signal because `ViewTilted` is the zero value of `ViewMode`. All
  session-scoped behaviour (auto-release, view restore, trail clear,
  zoom reset) gates on `LaunchSessionActive`. Manual `v` cycle into
  ViewLaunch from outside a session does **not** set the flag —
  player gets a standalone chase-cam view with no auto-release.
- **Route trigger is the active-slot Landed-false→true transition.**
  `World` carries two per-tick shadows: `wasActiveLanded bool` and
  `lastActiveCraft *Spacecraft` (pointer, not index — robust to undock
  slot-renumbering at `internal/sim/docking.go:330` where
  `ActiveCraftIdx--` shifts the index without changing the logical
  active craft). Each tick, before the ViewMode is read by the render
  layer:
  1. If `active != lastActiveCraft` (active-vessel switch — pointer
     diff): handle the switch (below) before the Landed-transition
     check. Special case: `lastActiveCraft == nil` (first tick after
     world init or save-load) snapshots only — no switch handler fires.
  2. If `active != nil && active.Landed && !wasActiveLanded`: fire
     `routeToLaunchView()` — store current ViewMode in `PrevViewMode`,
     set `LaunchSessionActive = true`, set `ViewMode = ViewLaunch`,
     clear `LaunchTrail`, reset `LaunchZoom = 0`, stamp
     `LaunchT0 = simTime`, zero `LaunchMaxQ`.
  3. Update `wasActiveLanded` and `lastActiveCraft`.

  Naturally covers Launchpad spawn (today's only path), future
  Touchdown semantics, and active-vessel-switch onto an already-
  Landed vessel.
- **Auto-release predicate: `LaunchSessionActive && apo > LaunchMissionFloorM`.**
  Restores `PrevViewMode`, clears `LaunchSessionActive` + `LaunchTrail`
  + `LaunchZoom` + `LaunchMaxQ` + `LaunchT0`, pushes a HUD toast.
  Hyperbolic / degenerate orbits (`a ≤ 0` or `e ≥ 1`) leave apoapsis
  undefined; auto-release simply doesn't fire — player stays in
  ViewLaunch until they cycle out, switch active, or the orbit becomes
  bound. Acceptable: a launch that's gone hyperbolic is in a state the
  player should be looking at anyway.
- **Manual `v` cycle out of ViewLaunch clears `LaunchSessionActive`** —
  player has taken over; no auto-release will fire even if apo
  eventually crosses the floor. ViewMode advances to the next mode in
  the cycle (ViewTilted), not back to `PrevViewMode` (cycle semantics
  are not "restore"). Re-entering ViewLaunch via the cycle gives a
  standalone session-less view.
- **Active-vessel-switch handling is four-case** (not two — the
  pre-grill two-case missed the "switch into a Landed vessel from no
  prior session" path):
  - *In session, new active Landed* → **hand-off**: keep
    `LaunchSessionActive`, keep `PrevViewMode`, clear `LaunchTrail` +
    `LaunchMaxQ` + `LaunchZoom`, **re-stamp `LaunchT0` to current
    sim-time**. Player sees the new vessel as if it had just spawned;
    T+ ticks from the switch moment.
  - *In session, new active not Landed* → **end**: restore
    `PrevViewMode`, clear all session state. Player gets their saved
    view back, focused on a non-Landed vessel.
  - *Not in session, new active Landed* → **fresh inline session**:
    same effect as a `routeToLaunchView()` call — save current ViewMode
    in `PrevViewMode`, set `LaunchSessionActive = true`, set
    `ViewMode = ViewLaunch`, stamp `LaunchT0`, clear trail / MaxQ /
    zoom. This is the path the pre-grill design missed.
  - *Not in session, new active not Landed* → **no-op** (nothing to
    do; player is moving between flying vessels with no launch state
    on either).
  - In every case, update `wasActiveLanded` to the new active's state
    *before* the Landed-transition check so we don't double-fire. When
    the switch handler opens or hands off a session, the post-handler
    Landed-transition check sees `Landed && wasActiveLanded` and
    correctly does nothing.
- **Auto-release is hard-cut with a HUD toast.** No animated camera
  transition — same beat as the ORBIT READY callout. Toast wording:
  `"ORBIT READY — returning to <prev view>"` (uppercase mirrors the
  v0.10.7 ORBIT READY callout the player already knows;
  `<prev view>` renders `PrevViewMode.String()`).
- **`ViewTilt.Theta` is suppressed inside ViewLaunch** (pure side
  view). Perspective tilt makes orientation reading *harder* at human
  scale because the viewer has to mentally un-project the tilt. ADR
  0002 captures the rationale; revisit only if playtest surfaces a
  use case.
- **Zoom: auto-altitude-driven with `+/-` manual override.** Default
  scale formula: `scale = max(1 m/cell, altitude / N_rows_above_horizon)`
  so the rocket stays centered and the horizon stays visible across
  the 3-order-of-magnitude altitude range (pad → 200 km).
  `+/-` pins scale until next Landed transition or auto-release.
  Independent of OrbitView's zoom — ViewLaunch carries its own scale
  state on `World.LaunchZoom`.
- **Projection-plane horizontal axis: live horizontal-of-commanded-
  attitude, fallback surface-frame east.** Each frame: project the
  commanded direction onto the local-horizontal plane; use it if
  `|horizontal| > epsilon`, else fall back to surface-frame east at
  the craft's current lat/lon (rotates with the body so the launch
  site stays anchored). No locked snapshot — cheap, deterministic,
  no extra state. Camera rotates smoothly as the player pitches over.
- **Trail is body-fixed, World-scoped, 1 s sim-time cadence, 256-
  point FIFO cap.** Cleared on auto-release AND on next Landed-entry
  transition AND on active-vessel-switch hand-off; manual `v` cycle
  out does **not** clear (player can cycle out to OrbitView mid-ascent
  and back to find the trail intact). Stored as
  `World.LaunchTrail []TrailPoint` where
  `TrailPoint = struct{ LatDeg, LonDeg, AltM float64; SampledAt time.Time }`.
  Render re-projects via
  `render.BodyFixedToWorld(body, lat, lon, simTime)` at current
  simTime so the trail visibly rotates with the body — the geographic
  launch site stays geographic. Reuses the same primitive the Landed
  integrator already uses (`internal/sim/landed.go:44`).
- **Horizon is always rendered as a curve** computed from camera
  altitude (body-silhouette circle in projection plane); below the
  horizon fills with `Body.SurfaceColor`. Same primitive at every
  scale — flattens to nearly-straight at the pad, bows visibly at
  altitude. No special-case threshold.
- **Body.SurfaceColor defaults to Body.Color** (additive catalog
  field, no save migration). Slice 1 ships the field and hand-sets
  Earth + Moon (the demo cases the player actually launches from);
  Slice 4 sweeps the rest of the catalog.
- **StageModule.LaunchSprite is an additive field on Stage** (no
  save migration). Slice 1 leaves it unset and renders the rocket as
  the existing `Glyph`; Slice 3 populates and switches.

## Slices

### v0.11.0 — ViewLaunch scaffolding + minimum visible scene

<!-- llm-parse: slice=v0.11.0 weight=M scope=sim+render+tui status=shipped tag=v0.11.0 -->

**Problem.** v0.10.7 yaw-anchored `ViewTilted` to local-vertical
during the launch phase, which fixes the "rocket slides sideways
across the screen" sub-problem but doesn't solve the deeper one:
orbit-scale rendering can't show the rocket, the launch tower, the
ground, the horizon curve, or the early trajectory shape at human
scale. Players want chase-plane orientation on launch with the
horizon at the bottom and the rocket ascending vertically — same
two reads the v0.10.7 grill identified ("orientation + early
trajectory") and that ADR 0002 commits to delivering through a
*separate* ViewMode rather than further hacks on the orbit view.

**Design.**

- **New `ViewLaunch` ViewMode enum entry** in `internal/sim/view.go`.
  Appended (not prepended) so existing zero-value default
  (`ViewTilted`) stays put — Slice 1 is for *new launch sessions*,
  not retroactively changing the default. Added to `AllViewModes`
  cycle order: `ViewTilted → ViewTop → ViewRight → ViewBottom →
  ViewLeft → ViewOrbitFlat → ViewLaunch → ViewTilted…`. No save
  migration (ViewMode is per-session UI preference, not persisted).
- **State machine fields on `World`** (none persisted):
  - `LaunchSessionActive bool` — the session sentinel.
  - `PrevViewMode ViewMode` — valid only when `LaunchSessionActive`.
  - `LaunchT0 time.Time` — sim-time at session start; HUD T+ anchor.
  - `LaunchMaxQ float64` — peak dynamic pressure across the session.
  - `LaunchTrail []TrailPoint` — body-fixed (lat, lon, alt, sampledAt)
    FIFO cap 256.
  - `LaunchZoom float64` — `+/-` override (0 = auto-zoom).
  - `wasActiveLanded bool` + `lastActiveCraft *Spacecraft` — per-tick
    shadows for the route trigger; pointer (not index) to survive
    undock slot-renumbering at `internal/sim/docking.go:330`. Not
    persisted; reset to `nil` on save-load so the first post-load
    tick snapshots without firing the switch handler.
- **Per-tick route/release handler** in `World.tick`, before the
  render layer reads ViewMode:
  1. If `active != lastActiveCraft` (pointer diff) and
     `lastActiveCraft != nil`: run the four-case switch handler
     (per Locked decisions above).
  2. If `active != nil && active.Landed && !wasActiveLanded`: fire
     `routeToLaunchView()`.
  3. If `LaunchSessionActive` and `apoAlt > LaunchMissionFloorM`
     (using the same predicate as v0.10.7's `LaunchAnchorPhi` ORBIT
     READY callout): fire `releaseLaunchSession()`.
  4. Update `wasActiveLanded = active.Landed` (or false if nil) and
     `lastActiveCraft = active`.

  Order matters: the active-switch handler runs first so the
  Landed-transition check uses the updated shadow. Auto-release runs
  after the route check so a same-tick Landed-and-immediately-apo>floor
  (theoretically impossible but cheap to handle correctly) doesn't
  thrash.
- **Manual `v` cycle while in ViewLaunch**: existing
  `World.CycleViewMode()` path; add a one-line guard — if leaving
  ViewLaunch via manual cycle, clear `LaunchSessionActive`. ViewMode
  advances to the next mode in the cycle (ViewTilted), not back to
  `PrevViewMode`. Re-entering ViewLaunch via the cycle gives a
  standalone session-less view (no auto-release; no trail clearing on
  next Landed-true tick — the route handler still fires on
  *transition*, not level).
- **ViewLaunch render path** in a new `internal/tui/screens/launch.go`
  (sibling of `orbit.go` — does not extend the orbit screen):
  - Compute camera: `cam = craft.State.R`. `local_up = (cam - body_center) /
    |…|`. `h_commanded = project(craft.commandedDir, plane=local_horizontal)`.
    `if |h_commanded| < eps: h_axis = body_east_at(craft_pos)` (surface-frame
    east, rotates with body). `else: h_axis = h_commanded / |h_commanded|`.
    Basis: `(h_axis, local_up)`.
  - Compute scale: if `LaunchZoom > 0`: `scale = LaunchZoom`. Else
    `scale = max(1.0, altitude / float64(canvas.Rows - horizonRows))`
    where `horizonRows` is the row-count below the rocket reserved
    for ground/horizon (e.g. canvas.Rows / 3).
  - Render scene:
    - **Horizon**: sample N points along the body-silhouette circle
      in the projection plane (radius = `R_body` projected through
      basis at camera position); draw as a curve. Below the curve,
      flood-fill with `Body.SurfaceColor` (falls back to `Body.Color`
      for bodies not yet populated; Earth + Moon hand-set in this
      slice).
    - **Pad marker**: body-fixed glyph at `(LaunchLatDeg,
      LaunchLonDeg)` of the active craft; transformed through
      Primary's current rotation phase. Drawn if depth-visible (near
      hemisphere, positive `(pad_world - cam) · DepthAxis`) and
      on-canvas. Glyph + colour chosen at implementation time
      (probably `+` in `ColorAccent`).
    - **Trail**: each point in `LaunchTrail` rendered as a faint dot
      at its body-fixed → world position (rotates with body). Same
      near-hemisphere depth check as the pad marker.
    - **Active vessel**: rendered as its existing `Glyph` at screen
      center. Sprite work is Slice 3.
  - Render HUD: bottom-strip overlay with `T+ HH:MM:SS  v_z ±XXX m/s
    | downrange X.X km  Q XX.X kPa (max YY.Y)`. Definitions:
    - **T+**: sim-time elapsed since the most recent Landed false→true
      transition (`World.LaunchT0` sim-time stamp captured by the
      route handler). Frozen at the auto-release value (or reset by
      the next Landed transition).
    - **v_z**: rate of change of altitude in the surface frame, m/s.
      Positive = climbing; negative = falling.
    - **downrange**: great-circle distance over the body surface from
      `(LaunchLatDeg, LaunchLonDeg)` to the craft's current sub-craft
      point, m → display as km.
    - **Q**: instantaneous dynamic pressure `0.5 · ρ · |v_rel|²` in
      kPa (Pa internally), where `ρ` is `Body.Atmosphere.Density(alt)`
      and `v_rel` is the air-relative velocity used by the drag
      integrator. Max-Q tracked across the launch session on
      `World.LaunchMaxQ`, reset by the route handler.
- **Trail sampling**: append `craft.bodyFixedPosition()` to
  `LaunchTrail` each tick if `simTime - lastSampleT >= 1.0 s` (or
  if buffer is empty). FIFO cap of 256.
- **Sprite suppressed: `ViewTilt.Theta` ignored inside ViewLaunch**
  — basis computation never reads it. ADR 0002 decision; documented
  inline so a future tilt-aware launch view doesn't accidentally
  re-enable it.
- **App-layer wiring** (`internal/tui/app.go`):
  - `App` gains a `launchView *screens.LaunchView` field alongside
    `orbitView`. Constructed in `NewApp` from the same theme.
  - `App.View()` default arm (today: `base = a.orbitView.Render(...)`)
    becomes: if `a.world.ViewMode == sim.ViewLaunch` →
    `a.launchView.Render(a.world, a.width, a.height)`; else the
    existing orbit path. ViewLaunch is presentational only — shares
    the same world, no separate `a.active` enum entry needed (the
    screen selection follows ViewMode).
  - `ZoomIn` / `ZoomOut` key handlers (`app.go:462-466`) need a
    ViewMode dispatch: if `a.world.ViewMode == sim.ViewLaunch` →
    `a.world.NudgeLaunchZoom(±)`; else the existing
    `a.orbitView.ZoomIn/Out()`. The world-side helper sets
    `LaunchZoom` to a scaled multiple of the current auto-scale
    (so the first `+` pins-and-zooms-in from the current view rather
    than snapping to a fixed value).
- **Chrome**: LaunchView is a minimal screen — title (`LAUNCH —
  <craft name>`), canvas (the chase-cam scene), HUD launch-readout
  strip overlaid at the bottom of the canvas, and a footer with
  ViewLaunch-specific key hints (`+/-` zoom, `v` cycle). **No
  sidepanel** — the orbit screen's body list / target panel are
  irrelevant during a launch session. Status messages overlay the
  last row exactly as on the orbit screen (`overlayLastLine` in
  `app.View()` works unchanged).

**Headline reads.** Two — **orientation** (the rocket is screen-up,
the ground is below, the horizon is the visible reference) and
**early trajectory** (the trail shows where the craft has come from
in the body-fixed frame; downrange shows ground-track distance).
Both are the reads ADR 0002 commits to.

**Weight.** M, sim + render + tui — comparable to v0.10.6's tilt
slice. No save migration (everything is per-session UI / world
state). Net-new render pipeline is the body-fixed sprite-at-lat-lon
and horizon-curve primitives in `widgets.Canvas`; both reusable for
Slices 2–4.

**Regression surface.** New: `internal/sim/launch_route_test.go`
(Landed-transition routes to ViewLaunch + stores PrevViewMode + sets
LaunchSessionActive; apo crosses 200 km releases + restores; manual
`v` cycle clears LaunchSessionActive without restoring view;
hyperbolic / `a≤0` post-Landed never auto-releases; active-vessel-
switch to Landed hands off + clears trail + re-stamps T0 when
already in session; active-vessel-switch to Landed opens a fresh
inline session when not previously in one (the four-case-handler
quadrant the pre-grill design missed); active-vessel-switch to
non-Landed ends session + restores view; active-vessel-switch
between two non-Landed vessels is a no-op; undock-renumber at
`docking.go:330` does NOT fire the switch handler (pointer-keyed
shadow ignores index-only changes); manual ViewLaunch entry
without a route never auto-releases). New:
`internal/sim/launch_trail_test.go` (cadence: 1 s sim-time; FIFO cap
holds at 256; clears on auto-release, Landed-entry, and active-
switch hand-off; survives manual `v` cycle out and back).
New: `internal/render/horizon_test.go` (curve point count + body-
frame silhouette math at varying altitudes). New:
`internal/tui/screens/launch_test.go` (HUD strip formatting; pad
marker depth-cull; ViewLaunch screen routing in App.View; zoom-key
dispatch by ViewMode). Existing orbit-screen + ViewTilted +
LaunchAnchor tests stay green (ViewLaunch is additive; ViewTilted's
basis path is untouched).

**Resolved (2026-05-24, two-pass grill in this doc).** First pass:
basic state-machine shape (edge-triggered route, manual escape),
zoom model (auto + `+/-`), projection-plane horizontal axis (live
commanded with surface-east fallback), HUD readout definitions
(T+, v_z, downrange great-circle, Q with max-Q), horizon curve
(always-curved, SurfaceColor fill), trail specifics (1 s, 256 cap).
Second pass: explicit `LaunchSessionActive` sentinel (PrevViewMode
zero-value collision fixed); per-tick `wasActiveLanded` shadow in
the route handler; active-vessel-switch behaviour (hand off if new
active Landed, else end + restore); LaunchZoom lives on World (not
the screen) for session-cohesion; LaunchView owns minimal chrome
(no sidepanel); trail stored as (lat, lon, alt, sampledAt) and
re-projected per render via the existing `render.BodyFixedToWorld`
primitive; App.View dispatches by ViewMode (no new `a.active` enum
entry); `ZoomIn`/`ZoomOut` key handlers gain a ViewMode dispatch.

**Resolved (2026-05-24, slice-time pre-impl grill, third pass).**

- **Switch detection is pointer-keyed.** `lastActiveCraft *Spacecraft`
  replaces `lastActiveIdx int`. The grep of `ActiveCraftIdx` mutations
  surfaced two bypass paths: undock renumber at
  `internal/sim/docking.go:330` (`w.ActiveCraftIdx--` shifts the index
  without changing the logical active) and direct assignment on save
  load at `internal/save/save.go:666`. Pointer comparison is robust
  to the renumber; save-load resets the shadow to `nil` and the first
  post-load tick snapshots without firing the switch handler.
- **Switch handler is four-case, not two.** The pre-grill design
  missed "switch into a Landed vessel from no prior session" — the
  Landed-transition step 2 wouldn't fire because step 4 already set
  `wasActiveLanded = true`, leaving the player in a Landed vessel
  with no session opened. The four-case shape (in-session × Landed,
  in-session × !Landed, !in-session × Landed → fresh inline session,
  !in-session × !Landed → no-op) covers every quadrant.
- **No save/load persistence of session state.** `LaunchSessionActive`,
  `LaunchT0`, `LaunchMaxQ`, `LaunchTrail`, `LaunchZoom` all stay
  per-session only. Player saves mid-launch, reloads, lands back in
  ViewLaunch with a fresh T+ (the persisted `Landed=true` re-fires
  the route handler on first post-load tick). Player saves mid-ascent
  (Landed already cleared) reloads into ViewTilted with v0.10.7's
  launch-anchor as the orientation fallback. Defer persistence until
  playtest surfaces a need.
- **Hand-off re-stamps `LaunchT0`** to current sim-time. Player
  switching active onto a freshly-Landed vessel sees T+ tick from
  the moment of switch, not from the original launch.
- **`NudgeLaunchZoom` step**: ×0.8 per `+`, ×1.25 per `-`
  (multiplicative, symmetric reciprocals so press-pairs return to
  origin). Floor 1.0 m/cell, ceiling unbounded. `LaunchZoom = 0`
  (auto) restored on session-end (auto-release, switch-end,
  hand-off, or fresh inline open).
- **HUD strip render layer**: overlay on the bottom braille row of
  the canvas (precedent: `overlayLastLine` on the orbit screen).
  Status messages overlay *on top of* the HUD strip when present,
  same precedent. No new screen-layout row.
- **Pad marker**: `+` in `ColorAccent`. Single-cell, visually

### v0.11.2 — body rotation convention unification

<!-- llm-parse: slice=v0.11.2 weight=L scope=render+physics+tui status=implemented adr=0003 -->

**Problem.** The v0.11.1 `/verify` surfaced that `render.BodyFixedToWorld`
(Snyder forward orthographic at ViewTop's sub-observer) does not rotate
body-fixed points about the same axis the integrator's `BodySpinOmegaWorld`
uses. For Earth (tilt 23.4°, AxialAzimuth 0) the axes are 32.6° apart;
spawn velocity `ω×r` and renderer-tracked surface positions diverge at
~250 m/s tangentially. Slice 2's 9-cell LUT made the gap visible; Slice
1's single-cell pad marker hid it. Slice 3 (stage sprites) and Slice 4
(polish) inherit the same mismatch on anything body-fixed. Full
diagnosis + alternatives in
[`docs/adr/0003-one-body-rotation-convention.md`](adr/0003-one-body-rotation-convention.md).

**Design.**

- **`render.BodyFixedToWorld` / `WorldToBodyFixed` rewrite.** Pure
  rotation of the body's body-fixed basis about `BodyRotationAxisWorld`
  by the rotation phase since `rotationEpoch`. Drop the
  `SubObserverPointDeg(CameraDirTop)`-derived Snyder construction; the
  function no longer depends on any view convention. Update the
  function doc to note: "agrees with the texture pipeline after the
  v0.11.2 unification of `projectPixelToLatLon`."
- **`render.projectPixelToLatLon` signature change.** Add a screen-up
  parameter expressing the canvas direction onto which the body's
  local-north at the sub-observer projects. Caller (each per-body
  painter, navball) computes it from the camera direction +
  `BodyRotationAxisWorld`. Inverse Snyder math then applies the same
  Gram-Schmidt screen rotation before mapping pixel → (lat, lon), so
  the painted texture orientation tracks the physical spin axis.
- **Per-body painter sweep.** 8 files in `internal/render/` (earth,
  mars, moon, jupiter, saturn, iceGiants, sun, galileans) update their
  `projectPixelToLatLon` calls to pass the screen-up vector. Painted
  texture orientation **changes** for any non-zero-tilt body — this is
  the corrected presentation; documented in the ADR's Consequences.
- **`lighting.go` + `navball.go`.** Same call-site update. Lighting is
  self-consistent (sub-solar lookup uses the same pixel→(lat,lon) as
  the painter, so the terminator stays in the right place); navball's
  texture overlay rotates the same way per-body painters do.
- **`bodyEpochOffsetDeg` re-sweep.** Per-body offsets tuned against
  the corrected projection so iconic features (Earth's
  Americas+Atlantic, Mars's Syrtis Major, Jupiter's Great Red Spot,
  Neptune's Great Dark Spot) still land at the visible disk centre
  for the canonical reference view. Offsets now express only the
  iconic-face choice — no embedded projection-correction term.
- **`physics.AtmosphereOmega` fold-in.** Replace the Z-aligned
  approximation with `render.BodySpinOmegaWorld(b)` (or a thin
  physics-package mirror so the physics package stays free of render
  imports). Drag direction for tilted-body vessels gains a small
  north/south component and a few-percent magnitude shift; the
  CONTEXT.md "engineering caveat" on Air-Relative Velocity is removed
  in the cycle-end vocabulary update.
- **Sim/spawn/landed/launch overlays.** **Zero source changes.** They
  continue to call `BodyFixedToWorld` and `BodySpinOmegaWorld`; the
  770 m drift dissolves automatically once the renderer agrees with
  the integrator.

**Headline reads.** The simulator has one rotation convention. A
future maintainer asking "which ω does X use?" has one answer.
Concretely for the player: Saturn V on KSC stays bolted to the LUT
through the gravity turn — the launch site stays geographically
anchored, the tower recedes at the rate the altitude actually
implies, the trail dots track the rocket's ground path. ViewTop on
Earth correctly displays the tilt (continents shift orientation
~32°; ViewRight shows the equator visibly edge-on with the spin axis
tilted off vertical).

**Weight.** L, render + physics + tui. Largest surface area in the
v0.11 cycle. No save migration (rotation conventions aren't
persisted). No public-type changes; only `projectPixelToLatLon` gains
a parameter (private to `internal/render/`).

**Regression surface.** Existing: every per-body painter's snapshot
tests regenerate (expected). `physics/drag_test.go` tilted-body
expected values regenerate. `internal/render/rotation_test.go`'s
`BodyFixedToWorld` round-trip tests must still pass — the inverse
relationship is preserved by construction. `internal/render/
navball_test.go` regenerates. New: `internal/render/rotation_test.go`
TestBodyFixedToWorldIsPureSpinAxisRotation — assert that for any
(lat, lon) and Δt, `BodyFixedToWorld(b, lat, lon, t+Δt)` equals
the rotation of `BodyFixedToWorld(b, lat, lon, t)` about
`BodyRotationAxisWorld` by `|BodySpinOmegaWorld| · Δt`. New:
`internal/sim/landed_test.go` TestLandedCraftStaysAtPadAfterLiftoff
— Saturn V on KSC, ignite, advance 3 s; assert
`|c.State.R - BodyFixedToWorld(KSC) · (R_earth + altitude)| <
1 m`. The same test that surfaces the bug today pins the fix.

**Resolved (2026-05-24, this doc's grill).**

- **Which side of the mismatch to align: renderer.** Sim/integrator
  stays on `BodyRotationAxisWorld` (the physical answer); renderer
  rewrites to agree. Alternatives considered + rejected in ADR
  0003.
- **Where it lands: own slice (v0.11.2).** Not folded into Slice 4 —
  weight is L, not S+. Slice 3 (stage sprites) and Slice 4 (polish)
  push back to v0.11.3 and v0.11.4 respectively.
- **ADR scope: wide.** One rotation convention, period — including
  `physics.AtmosphereOmega`, not just the renderer. Avoids leaving
  a third frame on the books for "future cleanup."
- **`bodyEpochOffsetDeg` sweep goal: iconic-face only, ship the
  corrected rotation.** Offsets re-tuned against the corrected
  projection so iconic features stay centred; the projection
  rotation itself presents as it physically should. Players see
  Earth rendered with the tilt correctly affecting ViewTop /
  ViewRight; release notes flag the change.
- **v0.11.1 release tactic: hold until v0.11.2 ships, then tag
  together.** Branch `v0.11.1-lut-soi` stays parked at `cfd843d`;
  v0.11.2 work continues on the same branch (or a fresh
  `v0.11.2-rotation-frame` branch that merges v0.11.1's commits
  forward — TBD at implementation time).

**Resolved at slice-open (2026-05-24, implementation pass).**

- **Mirror or import** for `physics.AtmosphereOmega`: **mirror**.

- **Snapshot regeneration policy**: no painter-snapshot diffs
  triggered in the sweep — the existing per-body painter tests use

  argument with default `(0, 1)` collapses `projectPixelToLatLon`
  to its pre-v0.11.2 behaviour for those cases. Visible orientation
  changes for tilted bodies on non-equator-on views are real but
  unpinned by the painter tests; visual verification deferred to
  `/verify`.

### v0.11.3 — stage-aware launch sprites

<!-- llm-parse: slice=v0.11.3 weight=M scope=spacecraft+render+tui status=shipped -->

**Scope (design-pass).** The rocket-as-Glyph from Slice 1 is a
placeholder; Slice 3 makes the rocket *look like its stack*:

- **New `StageModule.LaunchSprite` field** — per-stage ASCII art

- **Exhaust flame proportional to throttle** — below the firing
  stage (`Stages[0]`), a flame sprite scaled by `Throttle` (and
  hidden when `Throttle == 0` or engine is off). Pulsing optional
  polish.
- **Dropped stages render their own sprite** in scene (Slice 2's
  all-vessels-in-SOI pass picks them up; this slice gives them the
  composed-stack-of-one sprite instead of a single glyph).

Apollo-Stack visibly looks Apollo; custom stacks built via Slice
v0.10.1's stack configurator visibly reflect their actual
configuration.

**Weight.** M, spacecraft + render + tui. Field add + render path
work; ASCII sprite content for the v0.10.1 catalog parts.

**Resolved at slice-open** (`/grill-with-docs` 2026-05-25, revised
post-playtest 2026-05-25):

1. **Sprite primitive — braille pixels, not ASCII glyphs.** Each
   Stage paints a (`spriteWidthPx` sub-pixels wide × `LaunchSpriteRowsPx`
   tall) filled rectangle of braille dots via `widgets.Canvas.PlotColored`.
   No `SetCellOverlay` glyphs anywhere on the rocket render path;
   per-Stage identity reads from color + height, not silhouette
   characters. Braille dots are direction-agnostic, so a tilted
   rocket leans smoothly at any pitch — the gravity-turn smear the
   first-cut ASCII approach produced is gone.

   > **First-cut history (rejected post-playtest).** The original
   > slice-open grill picked ASCII glyphs (`╔╗ ║║ ▓▓ ╚╝` banded
   > columns, 2 cells wide, upright cells with stack-axis tilt). The
   > /verify playtest at altitude 29 km showed the rocket pitched to
   > ~84° from vertical: the 2-col-wide width axis swung perpendicular
   > to the near-horizontal stack, col 0 / col 1 cells landed on
   > different terminal rows, and consecutive sprite rows stepped to
   > new terminal columns. The result was a 30-cell-wide horizontal
   > smear of scattered glyphs, not a rocket. Box-drawing characters
   > have intrinsic orientation (`╔` vs `╗` vs `╚` vs `╝`) that
   > can't survive arbitrary projection. Replacing ASCII with braille
   > pixels removes both problems: pixels carry no orientation, and
   > the canvas's existing braille rendering accumulates dots into
   > each cell regardless of how they were plotted.

2. **Per-Stage shape — `LaunchSpriteRowsPx` height, constant width.**
   `Stage.LaunchSpriteRowsPx` is an `int` (braille sub-pixels) on
   the Stage struct. `spriteWidthPx = 2` constant — narrow constant
   width preserves the "thin rocket" reading and means the width
   axis is always small enough to never produce a smear, even at
   90° pitch. Per-Stage `LaunchSpriteRowsPx` values are *stylised*
   (chosen so proportions read right at pad-launch zoom), not real
   metres. Zero `LaunchSpriteRowsPx` falls back to the vessel-level
   `Glyph` render.

3. **Composition + tilt — pixels project naturally.** Each pixel sits
   at world offset
   `rowAbove · scaleMPerPx · cmd_screen + xPx · scaleMPerPx · width_screen`
   where `cmd_screen = (cmd · hAxis, cmd · localUp)` (unit-normalised)
   and `width_screen = (cmd_screen.y, -cmd_screen.x)` (perpendicular,
   rotated −90° in screen so col 0 lands LEFT of col 1 — pinned by
   `TestComposeLaunchSprite_Col0LeftOfCol1`). Foreshortening when
   `cmd·localUp < 1` falls out of the projection naturally.

4. **Flame — same braille primitive, gated on active burn.** Below
   `Stages[0]`'s base, an amber (`render.ColorWarning`) braille
   column of the same width as the rocket extends along
   `-cmd_screen`. Length-binned by throttle:
   - `Throttle ≤ 0` or no active burn: skip render entirely.
   - `0 < Throttle ≤ 1/3`: 4 sub-pixel rows (1 cell tall).
   - `1/3 < Throttle ≤ 2/3`: 8 sub-pixel rows.
   - `2/3 < Throttle`:       12 sub-pixel rows.

   Pulse: frame index `(time.Now().UnixMilli()/100) % 2`; frame B
   shifts the flame down by 1 sub-pixel so the dot pattern within
   each cell visibly repaints between frames at the ~100 ms cadence.
   Gating: flame renders only when `ManualBurn != nil ||
   ActiveBurn != nil` — pad-spawn loadout-default `Throttle=1.0`
   alone does not paint flame.

5. **LUT integration — overlay clear at rocket cells.** The LUT
   continues to use `SetCellOverlay` glyphs (`║ ╤ █`) for its
   tower-silhouette read; the rocket render calls
   `widgets.Canvas.ClearCellOverlay` per pixel cell after
   `PlotColored`, so braille dots show through at pad cells the LUT
   would otherwise mask. The LUT's right column (one cell east of
   the rocket at the pad) keeps its overlay glyphs intact and reads
   as "tower next to the rocket."

6. **Dropped-stage rendering — bare reuse, cmd preserved at decouple.**
   On stage jettison, `buildJettisonedCraft` sets
   `dropped.CurrentAttitudeDir = parent.CurrentAttitudeDir` (so the
   stage stays pitched at the angle it shed at — no jitter).
   Render path: `drawSOICraft` routes through the same composed-
   sprite path as the active vessel; dropped Spacecraft has
   `Stages = [popped stage]` only, so the composed render produces
   a single-stage braille silhouette. No flame (engine is off).
   No fade, no age field, no special despawn — free-fall integration
   arcs it back; `isNearHemisphere` culls below the horizon.

7. **LUT cell stride pinned to real-world metres (b73c54b).** The
   first-cut LUT (v0.11.1) and the v0.11.3 first cut both computed
   per-row stride as `scaleMPerPx · canvasCellPxH`, so the LUT's
   world height grew with the chase-cam autozoom — `scaleMPerPx`
   scales with altitude, so the tower scaled with the rocket and the
   rocket could never clear it (no real solution to
   `altitude < (4/3)·altitude`). Fix: `lutRowHeightM = 7.5 m`
   (~60 m total tower height, stylised from LC-39A's ~135 m crawler
   tower) and `lutColWidthM = 4 m` (8 m total LUT width — narrower
   than the real ~50 m square MLP since the silhouette is the
   visible-tower part, not the platform). The LUT now shrinks on
   screen as autozoom grows — correct perspective behaviour: the
   tower is a fixed real-world structure, the rocket recedes from
   it, the LUT becomes a few cells then disappears off-canvas as the
   rocket gains altitude. Pinned by
   `TestLaunchTowerRecedesAsRocketClimbs` (at altitude 200 m, the
   LUT crown `╤` must not appear above the canvas vertical centre;
   pre-fix LUT top would land at ~267 m, well above the rocket and
   above canvas centre).

  ---

  Surfaced during the v0.11.1 `/verify` on 2026-05-24. The
  integrator and the texture pipeline don't agree on what "the pad
  is at lat 28.6°N, lon -80.6°W at simTime t" means in world
  coordinates, so anything body-fixed (LUT, pad marker, trail dots)
  drifts away from the rocket as soon as the integrator takes over
  from `Landed=true`.

  **Symptom.** Saturn V on KSC, `b` to ignite, 3 sim-seconds in
  (altitude ~10 m): the pad's world position via
  `render.BodyFixedToWorld(lat, lon, simTime+3)` and the rocket's
  integrated `c.State.R` are ~770 m apart, almost entirely in the
  tangential direction. The radial component is the expected
  ~10 m altitude offset. The tangential 770 m offset is the bug —
  the rocket and the pad should be co-located in body-fixed frame
  to within Coriolis terms (~cm/s scale, not ~250 m/s).

  **Root cause.** `surfaceSpawnPosVel` seeds initial velocity with
  `render.BodySpinOmegaWorld` (the tilted physical spin axis ω),
  and `integrateLanded` continues to use that ω while Landed. Once
  `Landed=false` the physical integrator evolves the state under
  Newton's laws as a pure inertial trajectory tangent to ω×r.
  Meanwhile `render.BodyFixedToWorld(lat, lon, simTime)` uses a
  ViewTop-specific Snyder forward orthographic projection — the
  function comment is explicit that it is "internally self-
  consistent per view but cross-view inconsistent", and crucially
  it is **not a pure rotation around the spin axis** as simTime
  advances. The two evolutions diverge at ~250 m/s.

  **Why it didn't surface earlier.** Slice 1's pad marker is a
  single cell; the v0.10.7 launch-anchor read for ViewTilted lives
  at orbit scale where 770 m is sub-pixel; v0.10's launch HUD
  reads (T+, v_z, downrange, Q) don't depend on world-frame
  consistency between the integrator and `BodyFixedToWorld`. The
  v0.11.0 handoff framed the disappearing pad marker as a
  canvas-bounds story ("falls off the canvas top once altitude
  exceeds ~pxH/2 / scale_px") which pointed Slice 2 at making the
  sprite bigger rather than at the underlying frame mismatch.
  Slice 2's LUT (a 9-cell sprite) just makes the symptom obvious.

  **Two ways to fix, both architectural:**

  1. **Align `BodyFixedToWorld` to a pure spin-axis rotation.** The
     current ViewTop convention was chosen so the texture pipeline
     and the surface lat/lon agree on the orbit screen's most
     common view. Switching to physical rotation moves the texture
     work onto every consumer (orbit screen, lighting, eclipse,
     navball). Likely an ADR-sized change.
  2. **Move spawn velocity + landed integration into the
     `BodyFixedToWorld` convention.** Re-derive ω as the angular
     velocity of `BodyFixedToWorld(lat, lon, t)` over time, rather
     than the body's physical spin axis. Local to `sim/spawn.go`,
     `sim/landed.go`, and `spacecraft.surfaceSpawnPosVel`, but
     introduces a "convention ω" that diverges from
     `physics.AtmosphereOmega` (already Z-aligned for drag) — a
     third frame.

  **Where this lands.** Natural home is Slice 4 (v0.11.3) as a
  named "Edge cases" item — promotes the existing "edge cases"
  bullet from generic to specific. If the fix bottoms out as
  option (1), promote to its own ADR + a follow-up cycle. Either
  way needs a slice-time grill before implementation. **Do not
  attempt to compensate at the screen layer** (e.g. drawing the
  LUT relative to `c.State.R` instead of `BodyFixedToWorld`) —
  that would paper over the frame mismatch and break the trail's
  body-fixed re-projection, which currently depends on the
  one-frame-on-the-screen-layer invariant.

  Tracked here so Slice 3 doesn't get blindsided when stage
  sprites inherit the same mismatch. Slice 3's composed-stack
  rocket is drawn at `c.State.R` (integrator frame) so it is on
  the *correct* side of the disconnect; the dropped-stage sprites
  in Slice 3 will also live in the integrator frame because they
  spawn via `World.StageActive` which inherits `c.State.R` /
  `c.State.V` at decouple time. So Slice 3 doesn't add new
  surface area to the mismatch — but anything else body-fixed
  added in v0.11.x will.
