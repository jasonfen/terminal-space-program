# v0.11 — the launch experience grows up

<!-- llm-parse: cycle=v0.11 status=open anchor=launch-phase-fidelity -->

## Context

v0.10 ("the fleet flies for real") shipped v0.10.0 → v0.10.7 and
closed with the v0.10.6/.7 perspective-tilt + launch-anchor pair —
which exposed that orbit-scale rendering still can't deliver two
reads the launch phase needs: **early trajectory shape** (the
gravity-turn arc relative to the ground) and **orientation cues at
human scale**. v0.11 turns from *flying the fleet* to *growing up
the launch experience*: the headline is **launch-phase fidelity** —
a chase-cam scene at human scale that subsumes v0.10.7's
launch-anchor rendering while keeping its sim primitives load-bearing.

Workflow framing:

> Watch the rocket sit on the pad with the LUT beside it; light the
> engine and see the tower fall away; ride the chase-cam through the
> gravity turn with the horizon curving below; coast through the
> Karman line and watch the auto-release hand you back to the orbit
> view with an actual orbit-shape read once one exists.

That makes the cycle's spine **chase-cam scene pipeline →
launch-site detail (LUT + all-vessels-in-SOI) → stage-aware
launch sprites → polish/SurfaceColor sweep**. v0.10.7 keeps doing
its job on the orbit screen (anchored ascent-profile read); v0.11
adds the *human-scale* read that orbit-scale can't produce.

**Feature set locked** (2026-05-24) — v0.11 is launch-view-only;
mission scripting stays deferred to its own future cycle (same
posture as v0.10). The slice contract below is committed; subsequent
fold-ins need explicit approval, same as v0.10. Slice 1 is fully
designed (this doc); Slices 2–4 are at design-pass depth with the
headline observations from the ADR-0002 grill banked — each will
get its own slice-open grill before implementation, per v0.10
convention.

## Locked decisions

<!-- llm-parse: decisions=locked -->

The architectural decision lives in
[`docs/adr/0002-launch-view-as-distinct-viewmode.md`](adr/0002-launch-view-as-distinct-viewmode.md) —
ViewLaunch is a distinct ViewMode subsuming v0.10.7's anchor
*rendering* while keeping `sim.LaunchAnchorPhi` and
`sim.LaunchMissionFloorM` as reusable sim primitives. ViewTilted
keeps the anchor as its yaw rule (unchanged at orbit scale);
ViewLaunch is the new chase-cam read.

Slice 1 design decisions resolved 2026-05-24 (this doc's grill pass):

- **Session sentinel is explicit `World.LaunchSessionActive bool`.**
  `PrevViewMode` (a `ViewMode`) can't carry "no active session" as a
  signal because `ViewTilted` is the zero value of `ViewMode`. All
  session-scoped behaviour (auto-release, view restore, trail clear,
  zoom reset) gates on `LaunchSessionActive`. Manual `v` cycle into
  ViewLaunch from outside a session does **not** set the flag —
  player gets a standalone chase-cam view with no auto-release.
- **Route trigger is the active-slot Landed-false→true transition.**
  `World` carries two per-tick shadows: `wasActiveLanded bool` and
  `lastActiveIdx int`. Each tick, before the ViewMode is read by the
  render layer:
  1. If `activeIdx != lastActiveIdx` (active-vessel switch): handle
     the switch (below) before the Landed-transition check.
  2. If `active != nil && active.Landed && !wasActiveLanded`: fire
     `routeToLaunchView()` — store current ViewMode in `PrevViewMode`,
     set `LaunchSessionActive = true`, set `ViewMode = ViewLaunch`,
     clear `LaunchTrail`, reset `LaunchZoom = 0`, stamp
     `LaunchT0 = simTime`, zero `LaunchMaxQ`.
  3. Update `wasActiveLanded` and `lastActiveIdx`.

  Naturally covers Launchpad spawn (today's only path), future
  Touchdown semantics, and active-vessel-switch onto an already-
  Landed vessel.
- **Auto-release predicate: `LaunchSessionActive && apo > LaunchMissionFloorM`.**
  Restores `PrevViewMode`, clears `LaunchSessionActive` + `LaunchTrail`
  + `LaunchZoom` + `LaunchMaxQ` + `LaunchT0`, pushes a HUD toast.
  Hyperbolic / degenerate orbits (`a ≤ 0` or `e ≥ 1`) leave apoapsis
  undefined; auto-release simply doesn't fire — player stays in
  ViewLaunch until they cycle out, switch active, or the orbit becomes
  bound. Acceptable: a launch that's gone hyperbolic is in a state the
  player should be looking at anyway.
- **Manual `v` cycle out of ViewLaunch clears `LaunchSessionActive`** —
  player has taken over; no auto-release will fire even if apo
  eventually crosses the floor. ViewMode advances to the next mode in
  the cycle (ViewTilted), not back to `PrevViewMode` (cycle semantics
  are not "restore"). Re-entering ViewLaunch via the cycle gives a
  standalone session-less view.
- **Active-vessel-switch handling** (when `LaunchSessionActive`):
  - If new active vessel is Landed → *hand off*: keep
    `LaunchSessionActive`, keep `PrevViewMode`, clear trail / T0 /
    MaxQ (start a fresh launch session against the new vessel).
    Player sees the new vessel as if it had just spawned.
  - Else → *end*: restore `PrevViewMode`, clear all session state.
    Player gets their saved view back, focused on a non-Landed vessel.
  - In both cases, update `wasActiveLanded` to the new active's state
    *before* the Landed-transition check so we don't double-fire.
- **Auto-release is hard-cut with a HUD toast.** No animated camera
  transition — same beat as the ORBIT READY callout. Toast wording:
  *"orbit-ready — handing off to <prev view>"* (TBD wording at slice
  time).
- **`ViewTilt.Theta` is suppressed inside ViewLaunch** (pure side
  view). Perspective tilt makes orientation reading *harder* at human
  scale because the viewer has to mentally un-project the tilt. ADR
  0002 captures the rationale; revisit only if playtest surfaces a
  use case.
- **Zoom: auto-altitude-driven with `+/-` manual override.** Default
  scale formula: `scale = max(1 m/cell, altitude / N_rows_above_horizon)`
  so the rocket stays centered and the horizon stays visible across
  the 3-order-of-magnitude altitude range (pad → 200 km).
  `+/-` pins scale until next Landed transition or auto-release.
  Independent of OrbitView's zoom — ViewLaunch carries its own scale
  state on `World.LaunchZoom`.
- **Projection-plane horizontal axis: live horizontal-of-commanded-
  attitude, fallback surface-frame east.** Each frame: project the
  commanded direction onto the local-horizontal plane; use it if
  `|horizontal| > epsilon`, else fall back to surface-frame east at
  the craft's current lat/lon (rotates with the body so the launch
  site stays anchored). No locked snapshot — cheap, deterministic,
  no extra state. Camera rotates smoothly as the player pitches over.
- **Trail is body-fixed, World-scoped, 1 s sim-time cadence, 256-
  point FIFO cap.** Cleared on auto-release AND on next Landed-entry
  transition AND on active-vessel-switch hand-off; manual `v` cycle
  out does **not** clear (player can cycle out to OrbitView mid-ascent
  and back to find the trail intact). Stored as
  `World.LaunchTrail []TrailPoint` where
  `TrailPoint = struct{ LatDeg, LonDeg, AltM float64; SampledAt time.Time }`.
  Render re-projects via
  `render.BodyFixedToWorld(body, lat, lon, simTime)` at current
  simTime so the trail visibly rotates with the body — the geographic
  launch site stays geographic. Reuses the same primitive the Landed
  integrator already uses (`internal/sim/landed.go:44`).
- **Horizon is always rendered as a curve** computed from camera
  altitude (body-silhouette circle in projection plane); below the
  horizon fills with `Body.SurfaceColor`. Same primitive at every
  scale — flattens to nearly-straight at the pad, bows visibly at
  altitude. No special-case threshold.
- **Body.SurfaceColor defaults to Body.Color** (additive catalog
  field, no save migration). Slice 1 ships the field and hand-sets
  Earth + Moon (the demo cases the player actually launches from);
  Slice 4 sweeps the rest of the catalog.
- **StageModule.LaunchSprite is an additive field on Stage** (no
  save migration). Slice 1 leaves it unset and renders the rocket as
  the existing `Glyph`; Slice 3 populates and switches.

## Slices

### v0.11.0 — ViewLaunch scaffolding + minimum visible scene

<!-- llm-parse: slice=v0.11.0 weight=M scope=sim+render+tui status=designed -->

**Problem.** v0.10.7 yaw-anchored `ViewTilted` to local-vertical
during the launch phase, which fixes the "rocket slides sideways
across the screen" sub-problem but doesn't solve the deeper one:
orbit-scale rendering can't show the rocket, the launch tower, the
ground, the horizon curve, or the early trajectory shape at human
scale. Players want chase-plane orientation on launch with the
horizon at the bottom and the rocket ascending vertically — same
two reads the v0.10.7 grill identified ("orientation + early
trajectory") and that ADR 0002 commits to delivering through a
*separate* ViewMode rather than further hacks on the orbit view.

**Design.**

- **New `ViewLaunch` ViewMode enum entry** in `internal/sim/view.go`.
  Appended (not prepended) so existing zero-value default
  (`ViewTilted`) stays put — Slice 1 is for *new launch sessions*,
  not retroactively changing the default. Added to `AllViewModes`
  cycle order: `ViewTilted → ViewTop → ViewRight → ViewBottom →
  ViewLeft → ViewOrbitFlat → ViewLaunch → ViewTilted…`. No save
  migration (ViewMode is per-session UI preference, not persisted).
- **State machine fields on `World`** (none persisted):
  - `LaunchSessionActive bool` — the session sentinel.
  - `PrevViewMode ViewMode` — valid only when `LaunchSessionActive`.
  - `LaunchT0 time.Time` — sim-time at session start; HUD T+ anchor.
  - `LaunchMaxQ float64` — peak dynamic pressure across the session.
  - `LaunchTrail []TrailPoint` — body-fixed (lat, lon, alt, sampledAt)
    FIFO cap 256.
  - `LaunchZoom float64` — `+/-` override (0 = auto-zoom).
  - `wasActiveLanded bool` + `lastActiveIdx int` — per-tick shadows
    for the route trigger; not persisted, not part of the session.
- **Per-tick route/release handler** in `World.tick`, before the
  render layer reads ViewMode:
  1. If `activeIdx != lastActiveIdx`: handle the active-vessel
     switch (hand off vs end, per Locked decisions above).
  2. If `active != nil && active.Landed && !wasActiveLanded`: fire
     `routeToLaunchView()`.
  3. If `LaunchSessionActive` and `apoAlt > LaunchMissionFloorM`
     (using the same predicate as v0.10.7's `LaunchAnchorPhi` ORBIT
     READY callout): fire `releaseLaunchSession()`.
  4. Update `wasActiveLanded = active.Landed` (or false if nil) and
     `lastActiveIdx = activeIdx`.

  Order matters: the active-switch handler runs first so the
  Landed-transition check uses the updated shadow. Auto-release runs
  after the route check so a same-tick Landed-and-immediately-apo>floor
  (theoretically impossible but cheap to handle correctly) doesn't
  thrash.
- **Manual `v` cycle while in ViewLaunch**: existing
  `World.CycleViewMode()` path; add a one-line guard — if leaving
  ViewLaunch via manual cycle, clear `LaunchSessionActive`. ViewMode
  advances to the next mode in the cycle (ViewTilted), not back to
  `PrevViewMode`. Re-entering ViewLaunch via the cycle gives a
  standalone session-less view (no auto-release; no trail clearing on
  next Landed-true tick — the route handler still fires on
  *transition*, not level).
- **ViewLaunch render path** in a new `internal/tui/screens/launch.go`
  (sibling of `orbit.go` — does not extend the orbit screen):
  - Compute camera: `cam = craft.State.R`. `local_up = (cam - body_center) /
    |…|`. `h_commanded = project(craft.commandedDir, plane=local_horizontal)`.
    `if |h_commanded| < eps: h_axis = body_east_at(craft_pos)` (surface-frame
    east, rotates with body). `else: h_axis = h_commanded / |h_commanded|`.
    Basis: `(h_axis, local_up)`.
  - Compute scale: if `LaunchZoom > 0`: `scale = LaunchZoom`. Else
    `scale = max(1.0, altitude / float64(canvas.Rows - horizonRows))`
    where `horizonRows` is the row-count below the rocket reserved
    for ground/horizon (e.g. canvas.Rows / 3).
  - Render scene:
    - **Horizon**: sample N points along the body-silhouette circle
      in the projection plane (radius = `R_body` projected through
      basis at camera position); draw as a curve. Below the curve,
      flood-fill with `Body.SurfaceColor` (falls back to `Body.Color`
      for bodies not yet populated; Earth + Moon hand-set in this
      slice).
    - **Pad marker**: body-fixed glyph at `(LaunchLatDeg,
      LaunchLonDeg)` of the active craft; transformed through
      Primary's current rotation phase. Drawn if depth-visible (near
      hemisphere, positive `(pad_world - cam) · DepthAxis`) and
      on-canvas. Glyph + colour chosen at implementation time
      (probably `+` in `ColorAccent`).
    - **Trail**: each point in `LaunchTrail` rendered as a faint dot
      at its body-fixed → world position (rotates with body). Same
      near-hemisphere depth check as the pad marker.
    - **Active vessel**: rendered as its existing `Glyph` at screen
      center. Sprite work is Slice 3.
  - Render HUD: bottom-strip overlay with `T+ HH:MM:SS  v_z ±XXX m/s
    | downrange X.X km  Q XX.X kPa (max YY.Y)`. Definitions:
    - **T+**: sim-time elapsed since the most recent Landed false→true
      transition (`World.LaunchT0` sim-time stamp captured by the
      route handler). Frozen at the auto-release value (or reset by
      the next Landed transition).
    - **v_z**: rate of change of altitude in the surface frame, m/s.
      Positive = climbing; negative = falling.
    - **downrange**: great-circle distance over the body surface from
      `(LaunchLatDeg, LaunchLonDeg)` to the craft's current sub-craft
      point, m → display as km.
    - **Q**: instantaneous dynamic pressure `0.5 · ρ · |v_rel|²` in
      kPa (Pa internally), where `ρ` is `Body.Atmosphere.Density(alt)`
      and `v_rel` is the air-relative velocity used by the drag
      integrator. Max-Q tracked across the launch session on
      `World.LaunchMaxQ`, reset by the route handler.
- **Trail sampling**: append `craft.bodyFixedPosition()` to
  `LaunchTrail` each tick if `simTime - lastSampleT >= 1.0 s` (or
  if buffer is empty). FIFO cap of 256.
- **Sprite suppressed: `ViewTilt.Theta` ignored inside ViewLaunch**
  — basis computation never reads it. ADR 0002 decision; documented
  inline so a future tilt-aware launch view doesn't accidentally
  re-enable it.
- **App-layer wiring** (`internal/tui/app.go`):
  - `App` gains a `launchView *screens.LaunchView` field alongside
    `orbitView`. Constructed in `NewApp` from the same theme.
  - `App.View()` default arm (today: `base = a.orbitView.Render(...)`)
    becomes: if `a.world.ViewMode == sim.ViewLaunch` →
    `a.launchView.Render(a.world, a.width, a.height)`; else the
    existing orbit path. ViewLaunch is presentational only — shares
    the same world, no separate `a.active` enum entry needed (the
    screen selection follows ViewMode).
  - `ZoomIn` / `ZoomOut` key handlers (`app.go:462-466`) need a
    ViewMode dispatch: if `a.world.ViewMode == sim.ViewLaunch` →
    `a.world.NudgeLaunchZoom(±)`; else the existing
    `a.orbitView.ZoomIn/Out()`. The world-side helper sets
    `LaunchZoom` to a scaled multiple of the current auto-scale
    (so the first `+` pins-and-zooms-in from the current view rather
    than snapping to a fixed value).
- **Chrome**: LaunchView is a minimal screen — title (`LAUNCH —
  <craft name>`), canvas (the chase-cam scene), HUD launch-readout
  strip overlaid at the bottom of the canvas, and a footer with
  ViewLaunch-specific key hints (`+/-` zoom, `v` cycle). **No
  sidepanel** — the orbit screen's body list / target panel are
  irrelevant during a launch session. Status messages overlay the
  last row exactly as on the orbit screen (`overlayLastLine` in
  `app.View()` works unchanged).

**Headline reads.** Two — **orientation** (the rocket is screen-up,
the ground is below, the horizon is the visible reference) and
**early trajectory** (the trail shows where the craft has come from
in the body-fixed frame; downrange shows ground-track distance).
Both are the reads ADR 0002 commits to.

**Weight.** M, sim + render + tui — comparable to v0.10.6's tilt
slice. No save migration (everything is per-session UI / world
state). Net-new render pipeline is the body-fixed sprite-at-lat-lon
and horizon-curve primitives in `widgets.Canvas`; both reusable for
Slices 2–4.

**Regression surface.** New: `internal/sim/launch_route_test.go`
(Landed-transition routes to ViewLaunch + stores PrevViewMode + sets
LaunchSessionActive; apo crosses 200 km releases + restores; manual
`v` cycle clears LaunchSessionActive without restoring view;
hyperbolic / `a≤0` post-Landed never auto-releases; active-vessel-
switch to Landed hands off + clears trail; active-vessel-switch to
non-Landed ends session + restores view; manual ViewLaunch entry
without a route never auto-releases). New:
`internal/sim/launch_trail_test.go` (cadence: 1 s sim-time; FIFO cap
holds at 256; clears on auto-release, Landed-entry, and active-
switch hand-off; survives manual `v` cycle out and back).
New: `internal/render/horizon_test.go` (curve point count + body-
frame silhouette math at varying altitudes). New:
`internal/tui/screens/launch_test.go` (HUD strip formatting; pad
marker depth-cull; ViewLaunch screen routing in App.View; zoom-key
dispatch by ViewMode). Existing orbit-screen + ViewTilted +
LaunchAnchor tests stay green (ViewLaunch is additive; ViewTilted's
basis path is untouched).

**Resolved (2026-05-24, two-pass grill in this doc).** First pass:
basic state-machine shape (edge-triggered route, manual escape),
zoom model (auto + `+/-`), projection-plane horizontal axis (live
commanded with surface-east fallback), HUD readout definitions
(T+, v_z, downrange great-circle, Q with max-Q), horizon curve
(always-curved, SurfaceColor fill), trail specifics (1 s, 256 cap).
Second pass: explicit `LaunchSessionActive` sentinel (PrevViewMode
zero-value collision fixed); per-tick `wasActiveLanded` +
`lastActiveIdx` shadows in the route handler; active-vessel-switch
behaviour (hand off if new active Landed, else end + restore);
LaunchZoom lives on World (not the screen) for session-cohesion;
LaunchView owns minimal chrome (no sidepanel); trail stored as
(lat, lon, alt, sampledAt) and re-projected per render via the
existing `render.BodyFixedToWorld` primitive; App.View dispatches by
ViewMode (no new `a.active` enum entry); `ZoomIn`/`ZoomOut` key
handlers gain a ViewMode dispatch.

**Still open (slice-time, non-blocking).** Pad-marker glyph and
colour choice; HUD strip exact column layout and position;
auto-release toast wording; surface-frame-east degenerate case at
the body poles (latitude near ±90° makes "east" undefined — fall
back to a fixed body-X-axis projection? defer until a polar-launch
playtest surfaces it); zoom-key smooth-pin-from-current vs snap-to-
default (`NudgeLaunchZoom` implementation detail); whether the HUD
strip includes a mini-navball block or not (Slice 1 ships without;
revisit if playtest reads the launch view as nav-poor).

### v0.11.1 — launch-site detail: LUT tower + all-vessels-in-SOI

<!-- llm-parse: slice=v0.11.1 weight=M scope=render+tui status=designed -->

**Scope (design-pass).** Slice 1 establishes the scene pipeline;
Slice 2 fills it. Two additions, both reusing Slice 1's body-fixed-
sprite-at-lat-lon primitive:

- **Saturn-V LUT** (Launch Umbilical Tower) ASCII sprite, placed as
  a body-fixed sprite at the launch lat/lon. Visible whenever the
  player is at the pad (and after launch, recedes naturally as the
  rocket climbs). The tower stays anchored to the rotating body, so
  by the time the active vessel is at any altitude the tower has
  also rotated away from the camera plane.
- **All-Vessels-in-SOI rendering in scene.** ViewLaunch's render
  loop walks `World.Crafts` and includes any vessel in the active
  vessel's SOI within the projection bounds — so dropped stages
  (which spawn as their own passive vessels on decouple, per v0.9.1+)
  visibly fall away during staging. Reuses the body-fixed-glyph
  primitive; cull via the same near-hemisphere depth check.
- **Tower-falls-away cue** activates naturally: the LUT is body-
  fixed at the launch lat/lon; the rocket lifts and the camera
  follows the rocket, so the tower recedes into the projected
  scene. No animation needed — the body-fixed transform does the
  work.

**Weight.** M, render + tui. No sim change; no save migration. ASCII
sprite for the LUT is a content asset, not engineering work.
Reuses Slice 1's primitives entirely.

**Open at slice-open.** LUT sprite content (height, detail level,
single-tower vs full mobile-launcher); whether the LUT is per-
launch-site preset or shipped as a generic "tall structure at any
launch site" sprite; cull thresholds for dropped-stage rendering
(distance, age).

### v0.11.2 — stage-aware launch sprites

<!-- llm-parse: slice=v0.11.2 weight=M scope=spacecraft+render+tui status=designed -->

**Scope (design-pass).** The rocket-as-Glyph from Slice 1 is a
placeholder; Slice 3 makes the rocket *look like its stack*:

- **New `StageModule.LaunchSprite` field** — per-stage ASCII art
  (multi-line string, anchored at the stage's bottom centre).
  Additive on `Stage` (`json:",omitempty"`), no save migration.
  Catalog parts ship hand-authored sprites (S-IC big and squat,
  S-IVB narrower, CSM with conical command module).
- **Composed-from-stages rocket sprite** — render path concatenates
  the active vessel's `Stages` bottom-to-top, sprite per stage,
  composited into a single sprite. Rotated by `CurrentAttitudeDir`
  (so the rocket pitches over visibly as it gravity-turns).
- **Exhaust flame proportional to throttle** — below the firing
  stage (`Stages[0]`), a flame sprite scaled by `Throttle` (and
  hidden when `Throttle == 0` or engine is off). Pulsing optional
  polish.
- **Dropped stages render their own sprite** in scene (Slice 2's
  all-vessels-in-SOI pass picks them up; this slice gives them the
  composed-stack-of-one sprite instead of a single glyph).

Apollo-Stack visibly looks Apollo; custom stacks built via Slice
v0.10.1's stack configurator visibly reflect their actual
configuration.

**Weight.** M, spacecraft + render + tui. Field add + render path
work; ASCII sprite content for the v0.10.1 catalog parts.

**Open at slice-open.** Sprite anchor/scaling conventions (1 row =
N meters? per-stage independent? scale-with-zoom?); flame sprite
content + animation cadence; rotation discretisation (snap to N
angles or sub-cell smooth?); whether dropped-stage sprites
animate falling (gravity-arc visual) or just fade out.

### v0.11.3 — polish: stippled ground + SurfaceColor sweep + edge cases

<!-- llm-parse: slice=v0.11.3 weight=S scope=render+bodies status=designed -->

**Scope (design-pass).** Three trailing items:

- **Stippled ground-fill variant** — alternative to Slice 1's flat
  SurfaceColor fill, for low-altitude texture quieting (so the
  ground doesn't read as a solid bar). Same-hue stipple pattern
  (v0.10.6 convention from `internal/render/palette.go`); switches
  in below some altitude threshold or runs always.
- **Catalog population of `Body.SurfaceColor`** for every body —
  Slice 1 hand-sets Earth + Moon (the demo launch cases); Slice 4
  sweeps the rest of `systems/*.json`. Drives the horizon fill
  colour at each body.
- **Edge cases**:
  - **Surface-Contact crossings** (aerobraking vessel penetrates
    surface): ViewLaunch shouldn't blink on/off as the vessel
    bounces — the route handler should ignore Landed transitions
    that aren't *true* launch sessions (e.g. fuel == 0, or velocity
    above some surface-impact threshold). Heuristic TBD.
  - **No-active-vessel states**: ViewLaunch should fall back to
    OrbitView's default basis or push a "no active vessel" message,
    not crash.
  - **Polar-launch fallback** if not resolved in Slice 1.

**Weight.** S, render + bodies catalog work. Mostly content + polish,
no architectural surface.

**Open at slice-open.** Stipple pattern + threshold; SurfaceColor
values per body (visually grounded vs catalog-faithful); the exact
heuristic that distinguishes "true launch" from "bounced aerobraking
surface kiss" for the route-handler gate.

## Sequencing

Slice 1 is the foundation — every later slice consumes its scene
pipeline (body-fixed sprite/trail primitives, horizon, route/release
state machine, HUD strip). Slice 2 (LUT + all-vessels-in-SOI)
depends only on Slice 1's primitives; Slice 3 (stage-aware sprites)
depends on Slice 2's all-vessels-in-SOI pass (dropped stages render
their own sprites); Slice 4 is independent polish that can land any
time after Slice 1. Natural order: 1 → 2 → 3 → 4. Slice 1 must ship
and playtest before Slices 2–4 open for slice-time grilling —
vocabulary survival check, per project convention (CONTEXT.md
updates deferred until then).

## Deferred / out of scope for v0.11

- **Mission scripting / editor** — design-pass-first, its own future
  cycle (same posture as v0.10).
- **Multiplayer** (design current, foundations in, not slated).
- **Combined plane-shift + Hohmann** (the L-tier v0.10 deferred
  item, spec-committed in `docs/state-of-game.md`) — separate cycle
  candidate; not v0.11.
- **Powered landing / Touchdown semantics** (the differentiated
  Touchdown vs Crash outcomes; `CONTEXT.md` flags this is deferred).
  ViewLaunch is launch-focused; landing UX is its own future cycle.
- **Player-controlled `ViewTilt.Phi`** on the orbit view — v0.10.6
  deferred this pending a playtest signal; v0.11 doesn't revisit it
  (ViewLaunch has its own framing model, unrelated).
- **In-atmosphere drag overhaul** — Q is rendered as a HUD value;
