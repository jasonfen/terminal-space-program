# v0.8 — multi-craft polish

## Context

`docs/state-of-game.md` (snapshot at v0.7.6) frames v0.8 as an
unsorted bag of candidates in §2. This doc fleshes that out into
seven concrete slices organised around the v0.8 theme: **multi-craft
polish**.

The cycle's headline is "the sim grows up from a single rocket". By
the end of v0.8 the player can spawn multiple craft, each with a
distinct propulsion loadout and role; precision-maneuver them with a
monopropellant RCS thruster; dock them; reenter / aerobrake through
realistic Earth and Mars atmospheres; and watch the planets actually
rotate underneath them.

Multiplayer implementation, interstellar transfer, and N-body
perturbations are explicitly **deferred to v0.9+** — none of them
fit a "we don't have a craft that can barely make it to Mars" cycle.
Mission scripting is a stretch slice at the tail of v0.8 if there's
appetite, otherwise also v0.9.

v0.8.1 / v0.8.2 chain through the multi-craft save migration;
everything else is independent and can ship in any order.

## Resolved scoping questions

Settled before writing this plan:

1. **Theme.** "Multi-craft polish". Multiplayer implementation,
   interstellar (real cruise math or deus-ex jump), and N-body
   perturbations all defer to v0.9+ — the v0.8 cycle wants the
   *sim* to grow before the network or the universe does.
2. **Multi-craft spawn UX (v0.8.1 / v0.8.2).** Two paths ship:
   (a) keystroke `N` opens a spawn dialog letting the player pick
   craft type + initial fuel + orbital parameters + parent body;
   (b) staging — the active craft sheds a stage and the jettisoned
   stage becomes its own controllable craft. (d) mission-reward
   spawn flagged as future "story mode" but not in v0.8.
3. **Craft types — three axes.** All three of (i) propulsion
   loadout, (ii) role, (iii) visual differentiation ship in v0.8.2.
   Visual sidecar: engines firing (animated flame on the active-burn
   craft) + RCS puffing (per-thruster discharge marker) so the player
   can see *which* craft and *which* engine is active.
4. **Active-craft selector.** `[` / `]` cycle + click-to-select on
   the orbit canvas. Numbered slots (`1`–`9`) deferred until a save
   has > 4 craft routinely.
5. **Save schema (v0.8.1).** First non-additive migration — `Craft
   *Craft` → `Crafts []Craft`. Land the typed-migration machinery
   (per-version `vNtoNplusone` functions + versioned `payloadV4` /
   `payloadV5` types + a `testdata/save-v1.json` … `save-v4.json`
   round-trip corpus) as part of this slice. The `omitempty` pattern
   doesn't survive the field shape change.
6. **RCS sequencing (v0.8.0 first).** Ships **before** multi-craft.
   The monopropellant pool, mode toggle, and pulse model land
   against the single existing craft; multi-craft v0.8.1 then
   inherits RCS for free across all spawned craft. Cleaner than
   bundling the two.
7. **RCS pulse model.** Tap = single fixed-quantum pulse (e.g.
   0.1 m/s); hold = repeated pulses at fixed cadence. More
   physically realistic than a continuous low-thrust burn for
   sub-m/s precision work.
8. **RCS tuning.** Linear-with-mass: `RCSThrust = k_T · m_dry`,
   `Monoprop_capacity = k_M · m_dry` so larger craft get
   proportionally larger RCS budgets. Total budget tuned **higher**
   than the original §2 sketch (target ~30 m/s on a default-loadout
   craft, vs the original ~6 m/s) so docking proximity ops aren't
   twitchy. `RCSIsp ~220 s` stays per the §2 sketch. **v0.8.0
   tuning correction:** the initial cut sized monoprop at 50 kg
   based on the formula `Isp · g₀ · ln(m₀/m_dry)`, which conflated
   total fuel ejection with monoprop-only ejection. The realised
   budget at 50 kg was ~2 m/s, not the projected 28. Fixed in the
   v0.8.0 ship cut: capacity bumped to ~720 kg (~1.4 % wet mass),
   delivering the planned ~30 m/s. Sizing constant is `kCap = 720
   / 11000 ≈ 0.0654 kg/kg-dry`. Whether 30 m/s is *enough* for
   actual docking maneuvers is queued as an open question (see
   below) — playtest may push us to ~50–80 m/s.
9. **Docking model (v0.8.3).** State-transition stub: when two craft
   are within `dockingDist` (e.g. 50 m) at relative speed below
   `dockingV` (e.g. 0.1 m/s), they fuse into a single composite
   craft with summed mass, summed fuel + monoprop pools, and pooled
   Δv budget. Full proximity-ops simulation deferred. Multiplayer
   (v0.9+) may upgrade this.
10. **Atmospheric drag (v0.8.4).** As physically realistic as
    achievable without modelling craft shape or aerodynamic heating.
    Earth and Mars get scale-height exponential atmospheres tuned to
    real values; predictor is drag-aware so the dashed line
    inside an atmosphere doesn't lie. **Always-on**, not opt-in.
    Heating / structural overstress / aerodynamic-coefficient
    modelling defers to v0.9+.
11. **Atmospheric visual (v0.8.4).** Bodies with atmospheres render
    a graphical haze layer around the solid disk — limb glow on
    Earth (blue tier), Mars (rust tier), Venus / Titan added when
    those bodies get atmospheres. Falls into the same dispatch hook
    as the v0.7.2.x texture pipeline.
12. **Sim-time rotation (v0.8.5).** Picked over solar lighting +
    terminator (option A in the scoping pass) because A needs
    per-cell ANSI 24-bit mixing — `lipgloss` doesn't support
    per-pixel brightness, so A's research pass blocks ship cadence.
    Sim-time rotation threads `lon0(t)` through the existing
    `TextureFor` dispatch and is cheap. Tidally locked moons
    (Luna's far side from a heliocentric perspective; Galileans
    from Jupiter-frame views) need explicit handling so a locked
    moon viewed from outside its parent's frame still rotates
    naturally.
13. **Solar lighting + terminator + eclipses (deferred).** Option A
    plus eclipses fall out together — eclipses are nearly free once
    lighting lands. Both deferred to v0.9 with an explicit
    research-first item: investigate canvas-level ANSI 24-bit
    mixing as a `lipgloss` workaround before slicing the feature.
14. **Controls polish picks (v0.8.6).** Three items from the §2
    backlog: (b) `IterateForTarget` toggle in the `m` form, (c)
    multi-rev porkchop keys (`g` flip retrograde, `[`/`]` cycle
    `nRev`), (d) throttle-change warp clamp. (a) drag-to-edit
    nodes, (e) attitude-mode save persistence, (f) per-craft
    keymap chord stay deferred / fold into multi-craft. All v0.8.6
    decisions are reopenable later.
15. **Cadence.** Foundation-first: RCS → multi-craft → craft types →
    docking → drag → visual → controls polish. Ships the headline
    items (multi-craft, RCS, docking, drag) early; visual and
    controls polish trail.

## Slices

### v0.8.0 — RCS / monopropellant mode

Ships the precision-maneuver thruster against the **single existing
craft**. Multi-craft v0.8.1 inherits RCS unchanged.

**Code surface.**

- `internal/spacecraft/spacecraft.go` — new fields:
  - `Monoprop float64` (kg).
  - `MonopropCapacity float64` (kg).
  - `RCSThrust float64` (N total) — sized linearly off `DryMass`.
  - `RCSIsp float64` (~220 s).
- `internal/spacecraft/thrust.go` — `BurnMode` reused unchanged
  (six directions). Effective thrust + Isp + propellant pool
  picked at fire-time by a new `EngineMode` enum
  (`EngineMain` / `EngineRCS`).
- `internal/sim/world.go` — new `World.EngineMode` field. Mode
  toggle key (`r`) flips between main and RCS. HUD shows the
  armed engine.
- `internal/sim/maneuver.go` — `ManualBurn` integrator path
  branches on `EngineMode`: main goes through existing RK4 thrust;
  RCS dispatches to a new pulse-quantum integrator.
- New `internal/physics/rcs.go` — `StepRCSPulse(state, mass, mode,
  dvQuantum) -> state'`. Single-step state update for one pulse;
  no sub-stepping needed at the cm/s scale.
- `cmd/.../keys.go` — `r` cycles `EngineMode`. Attitude keys in
  RCS mode fire pulses on tap (single quantum) or sustained pulses
  on hold (cadence ~5 Hz).
- HUD: PROPELLANT block extends to show monoprop level. ATTITUDE
  block shows `engine: main | rcs`.
- Save schema: `Spacecraft.Monoprop` carries through with omitempty
  (transient state). v3 saves load with full monoprop.

**Visual sidecar.** Each RCS pulse stamps a short-lived puff marker
(2–3 ticks) at the craft position offset along the burn-mode
direction. Not the full v0.8.2 RCS-puffing visualisation — just
a placeholder that v0.8.2 replaces with per-thruster markers once
craft visual differentiation lands.

**Tuning numbers (default S-IVB-1 craft, 11000 kg dry + 40000 kg fuel,
shipped v0.8.0).**

- Monoprop capacity: **720 kg** (1.4 % of wet mass — bumped from
  the planning sketch's 50 kg, which gave only ~2 m/s budget; the
  scoping formula `Isp · g₀ · ln(m₀/m_dry)` conflated total fuel
  ejection with monoprop-only ejection).
- RCSThrust: ~440 N (linear scaling — kept from sketch).
- RCSIsp: 220 s.
- Δv budget: ~30 m/s (Δv = Isp · g₀ · ln(m_total / m_total_minus_monoprop)).
- Per 0.1 m/s pulse: ~2.4 kg of monoprop. Tank lasts ~300 pulses.

Tunable per-loadout in v0.8.2.

**Estimate.** ~400 LOC + tests. **Actual.** ~750 (code + new tests + corpus).

### v0.8.1 — multi-craft foundation

Promotes `World.Craft` from single pointer to a slice + active-craft
index, with the first non-additive save migration in the project.

**Code surface (sim).**

- `internal/sim/world.go` —
  - `Crafts []*Spacecraft` replaces `Craft *Spacecraft`.
  - `ActiveCraftIdx int`.
  - `World.ActiveCraft() *Spacecraft` accessor.
  - Audit every `w.Craft` read in the codebase (~40 sites
    estimate); rewrite as `w.ActiveCraft()` or per-craft loop
    where the operation should apply to all (e.g. integrator,
    SOI checks).
- `World.Tick` — integrator loop walks `Crafts`, propagating each
  in its own primary's frame. Existing logic already handles
  multi-frame propagation per-craft.
- Mission evaluator runs against `ActiveCraft()` only — per-craft
  mission attribution defers to v0.9+ alongside MP open question
  #3.
- Predictor walks `ActiveCraft()`'s nodes only; other craft show
  current-orbit ellipse but no chained predictions in v0.8.

**Code surface (selector).**

- `[` / `]` keys cycle `ActiveCraftIdx`. Wraps both ends.
- Click cascade extends: click a craft glyph → activate that
  craft (slots in priority above body / node / canvas).
- Title bar shows `CRAFT 1 / 3` chip when `len(Crafts) > 1`.
- HUD VESSEL block carries craft name; PROJECTED ORBIT chains
  the active craft only.

**Code surface (spawn).**

- `n` keystroke (or `N` if `n` keeps quick-plant) opens a
  `screens.SpawnCraft` form:
  - Craft type (cycle): default S-IVB-1 / ICPS / pure-RCS-tug
    (the v0.8.2 set, with single-default fallback if v0.8.1
    ships before v0.8.2).
  - Initial fuel (% of capacity).
  - Orbit: parent body cycle (any body in active system),
    altitude (km), inclination (°), prograde / retrograde.
- Form planted as `World.SpawnCraft(SpawnSpec)` returning the new
  `Spacecraft`. Active idx swaps to the new craft.

**Code surface (save migration).**

- `internal/save/save.go` — split into version-suffixed payload
  types:
  - `payloadV4` (current) — has `Craft *Craft`.
  - `payloadV5` (new) — has `Crafts []Craft`, `ActiveCraftIdx`.
  - Migration function `v4to5(p4 *payloadV4) *payloadV5` —
    wraps the single craft into a one-entry slice with
    `ActiveCraftIdx = 0`.
  - Forward chain: `loadEnvelope` reads version field, chains
    through migrations until current.
- `internal/save/testdata/save-v1.json` … `save-v4.json` corpus
  — frozen sample saves at each historical schema. Test:
  `TestLoadCorpus` round-trips each through current load.
- Schema bump: `version: 5`.

**Estimate.** ~700 LOC across sim/save/tui + tests + corpus.
Largest slice in v0.8.

### v0.8.2 — craft types

Three axes of craft differentiation, with engine-firing and RCS-
puffing visuals as the visible payoff.

**Code surface (propulsion loadouts — axis i).**

- `internal/spacecraft/loadouts.go` — new package-level catalog
  of named loadouts. v0.8 launch set:
  - `S-IVB-1`: J-2 (1023 kN, Isp 421, 11000 kg dry + 40000 kg
    fuel) — current default.
  - `ICPS`: RL-10 (108 kN, Isp 462, 3500 kg dry + 25000 kg
    fuel) — low-TWR, returns from v0.5.6.
  - `RCS-tug`: pure-monoprop (0 main thrust, 200 kg dry +
    100 kg monoprop) — for proximity ops.
  - `Lander`: lower thrust + ablative legs (TBD numbers; placeholder
    in v0.8.2, tuned in v0.8.x patches).
- `Spacecraft.LoadoutID string` — references the catalog entry.
  Loadout fills in `Thrust`, `Isp`, `RCSThrust`, etc. on spawn.

**Code surface (roles — axis ii).**

- `Spacecraft.Role string` — free-form tag (`transfer-stage` /
  `lander` / `orbiter` / `station`). Doesn't affect physics in
  v0.8 — surfaces in HUD / mission predicates only. Earmark for
  mission scripting (v0.8.7+ stretch or v0.9+).

**Code surface (visual differentiation — axis iii).**

- `Spacecraft.Glyph string` (e.g. `▲` / `◆` / `●`) — overrides
  the default chevron / disk on the orbit canvas.
- `Spacecraft.Color string` (hex) — overrides
  `ColorCraftMarker` for non-active craft. Active craft keeps
  the bright marker treatment.
- Trail color follows `Spacecraft.Color` per craft — the v0.5.2
  ring buffer becomes per-craft.

**Code surface (engine-firing visual).**

- `internal/render/flame.go` — new flame primitive: 3–6 px chevron
  trailing the craft chevron, oriented anti-velocity (or anti-
  thrust direction during a burn), in `ColorFlame` (warm
  yellow/orange tier). Pulses every render tick (animation).
- Triggered when craft has `ActiveBurn != nil` or `ManualBurn !=
  nil` and `Throttle > 0` and `EngineMode == EngineMain`.
- Hidden during RCS-only burns (RCS uses puff visualization
  below).

**Code surface (RCS-puffing visual).**

- `internal/render/rcs.go` — per-pulse marker. Each RCS pulse
  draws a 1–2 px puff at the craft offset along the pulse
  direction, fading over 4–6 render ticks. Position offset by
  thruster geometry (placeholder: prograde pulse → puff aft,
  normal+ pulse → puff to one side, etc.).
- Replaces v0.8.0's placeholder puff marker.

**Code surface (staging spawn).**

- New `S` keystroke (or `Ctrl+S` if `S` is taken — `S` is
  currently Save) — current craft sheds its lowest stage.
- `Spacecraft.Stages []Stage` — new field. Each stage has
  dry-mass, fuel-mass, thrust, Isp, monoprop pool. Default
  craft has one stage; staging craft (S-IVB-1 / ICPS) defined
  with multiple.
- Staging: pop bottom stage off `Stages`, spawn it as a new
  craft in the same orbit (with whatever fuel / monoprop
  remained), retain the upper stages on the active craft.
- Active-craft idx stays on the **upper** craft post-staging
  (the player's "main" craft); the dropped stage becomes a
  passive object the player can `[` cycle to.

**Estimate.** ~600 LOC across spacecraft/render/save + tests.

### v0.8.3 — docking state-transition stub

Two craft within proximity at low relative speed fuse.

**Code surface.**

- `internal/sim/docking.go` — per-tick proximity check across
  all craft pairs in the same primary frame:
  - `dist < dockingDist` (default 50 m).
  - `|v_rel| < dockingV` (default 0.1 m/s).
  - Both have a fuel / dock authorization bit (TBD: just always
    yes in v0.8, gated later).
- On match: fuse via `World.DockCrafts(idxA, idxB)`:
  - New craft's mass = sum of dry + fuel + monoprop masses.
  - Capacity sums (fuel + monoprop tanks pooled).
  - Δv budget recomputed via rocket equation against the
    **active** craft's main engine (the docked composite picks
    one engine to use; if neither has main, RCS-only).
  - Position = mass-weighted centroid; velocity = momentum-
    conserving combination.
  - Roles concatenated (`transfer-stage+lander`).
  - Trail / glyph / color from the active partner; the docked
    partner is removed from `Crafts` slice.
- Reverse — undocking — fires on player keystroke (`U`?). Splits
  back along stage boundaries if the docked composite has them;
  otherwise prompts player to assign mass between the two halves.
- Save schema: docked composites are just regular `Spacecraft`
  entries with summed masses; the dock event is not persisted
  (state, not history).

**Open scoping for v0.8.3 implementation.**

- How does the docking proximity check perform with > 5 craft?
  O(n²) is fine for v0.8 (small n); may need spatial hashing in
  v0.9+ MP / station-keeping scenarios.
- Visual: a brief "DOCKED" HUD flash on the moment of fusion?
  Or silent (player notices via the craft list shrinking)?

**Estimate.** ~250 LOC + tests.

### v0.8.4 — atmospheric drag

Realistic Earth + Mars exponential atmospheres; predictor and
live integrator both drag-aware; haze rendered around bodies
with atmospheres.

**Code surface (physics).**

- `internal/bodies/body.go` — new optional struct field:

  ```go
  type Atmosphere struct {
      ScaleHeight       float64 // m
      SurfaceDensity    float64 // kg/m³
      CutoffAltitude    float64 // m above surface — drag = 0 above
      Color             string  // hex, for haze rendering
  }
  type CelestialBody struct {
      // ...existing fields
      Atmosphere *Atmosphere `json:"atmosphere,omitempty"`
  }
  ```

- `internal/bodies/systems/sol.json` — populate Earth + Mars:
  - Earth: scale height 8500 m, surface density 1.225 kg/m³,
    cutoff 150 km, haze color light blue.
  - Mars: scale height 11100 m, surface density 0.020 kg/m³,
    cutoff 100 km, haze color rust-orange.
- `internal/physics/drag.go` — `DragAccel(r, v, primary,
  craft) float64` returns drag deceleration vector:
  - Density `ρ(h) = ρ_0 · exp(-h/H)` for `h < cutoff`, else 0.
  - Drag accel `a = -0.5 · ρ · |v_rel|² · C_D · A / m · v̂_rel`.
    `C_D · A / m` lumped as a `Spacecraft.BallisticCoefficient`
    field (default ~0.01 m²/kg for the S-IVB-1; tuned later).
  - `v_rel = v - v_atmosphere` where `v_atmosphere = ω_body × r`
    (atmosphere co-rotates with body — Earth's atmosphere isn't
    static).

**Code surface (integrator wiring).**

- `physics.StepVerlet` and `physics.StepRK4ThrustState` both
  accept an optional `extraAccel func(r,v) Vec3` parameter. Drag
  passes itself in.
- `World.Tick` builds the extra-accel closure once per craft per
  tick using the craft's primary as reference.
- Kepler warp-lock retreat: when craft is below atmospheric
  cutoff altitude of any body, `KeplerStep` is rejected — falls
  back to Verlet sub-stepped integration. This re-introduces the
  v0.4.3-era cost on atmospheric-skim trajectories, but only on
  the small fraction of orbits that touch atmosphere.

**Code surface (predictor coherence).**

- `PredictedSegments` already calls into the same `stepCraft`
  primitive per `integration-design.md` §5. Extending the closure
  to include drag automatically extends the predictor too.
- Sample density: keep the v0.7-era 96-sample horizon for now;
  the deferred adaptive-sampling work (`integration-design.md`
  §10) becomes more pressing once drag breaks Kepler warp-lock
  on close-approach trajectories.

**Code surface (haze rendering).**

- `internal/render/atmosphere.go` — `BodyAtmosphereOuterPx(b,
  scale)` returns the haze layer outer radius.
- Haze rendered as a faint outline ring at outer-altitude
  (= cutoff + ~scale-height), in `Atmosphere.Color` at low
  alpha (8-bit ANSI dimmed).
- Fades visibility as `pxRadius` of body grows past ~20 px
  (the haze layer should read as a halo, not as another disk).
- Falls into the same `TextureFor` dispatch chain as v0.7.2.x
  textures.

**Estimate.** ~500 LOC + JSON edits + tests.

### v0.8.5 — sim-time planet rotation + tidal-lock perspectives + textured-bodies trickle

Threading `lon0(t)` through the existing texture pipeline so the
surfaces actually spin.

**Code surface (rotation).**

- `bodies.CelestialBody.SiderealPeriod float64` (s) — already on
  some bodies; populate the rest.
- `bodies.CelestialBody.TidallyLocked bool` — populate Luna and
  the Galileans.
- `render.SubObserverLongitude(b, t) float64` —
  - Free body: `(2π · t / SiderealPeriod) mod 2π` plus a
    per-body epoch offset (so Earth's prime meridian doesn't sit
    at sub-observer-zero forever).
  - Tidally locked: `lon` of parent's anti-direction at time `t`
    — i.e. the body's "near side" always points at its parent.
    From outside the parent's frame (heliocentric view of Luna),
    this means the near side rotates as the parent moves; the
    moon itself doesn't spin in inertial space, but its visible
    face from a third-party observer changes.
- `render.{Earth,Mars,Jupiter,Moon}PixelColor` signatures extend
  with a `lon0` parameter; each body's ellipse table shifts
  accordingly.
- `render.TextureFor(b, r, lon0)` becomes the new dispatch
  entry; existing call sites pass in `SubObserverLongitude(b,
  world.Clock.SimTime)`.

**Code surface (warp clamp on rotation).**

- At very high warp (10000×+), per-render-tick `lon0` advance is
  large — Earth visibly spins under the camera. Two policies:
  - (a) clamp rotation rate at high warp (planet looks "frozen"
    above some threshold).
  - (b) let it spin (intentional kinetic feedback).
- v0.8.5 ships **(b)**, with a follow-up patch slot for (a) if
  it reads as distracting in playtest.

**Code surface (textured-bodies trickle — independent patches).**

- Saturn (with cloud bands + ring lighting tie-in): `mars.go`-
  shaped structure, target v0.8.5.x patch.
- Galilean moons (Io / Europa / Ganymede / Callisto): one body
  per patch, like v0.5.11 / .12 cadence.
- Uranus / Neptune: pale disks → banded.
- Each is a small patch on the v0.8.5 line, no architectural
  cost.

**Estimate.** ~300 LOC for the rotation core + ~200 LOC per
textured-body patch.

### v0.8.6 — controls polish bag

Originally three items (b/c/d) folded into one slice for ship
cadence. v0.8.6 in-flight scope expanded to a fourth (a) — a
keymap pass — after a playtest report that several uppercase /
lowercase pairs did unrelated things.

**Code surface.**

- **(a) Keymap pass.** Resolved case-collisions where uppercase
  and lowercase keys did unrelated things in the same mode, and
  pulled bindings closer to KSP defaults where reasonable.
  - Save / Load: `S` / `L` → **F5 / F9** (KSP quicksave /
    quickload). Drops the s/S (retrograde / save) and l/L
    (next-body / load) collisions outright.
  - ClearNodes (`N`): global keybinding retired. The case-
    collision with `n` (SpawnCraft) made remembering "spawn vs
    wipe" a constant trip-up. Replaced with two
    maneuver-form actions:
    - `ctrl+d` (in `m` form) — delete the planted node being
      edited (no-op when creating new). Emits `NodeDeleteMsg`.
    - `ctrl+k` (in `m` form) — clear ALL planted nodes for the
      active craft. Emits `NodeClearAllMsg`.
  - New sim API: `World.DeleteNode(idx int)` with out-of-range
    no-op behaviour. Tests cover happy + boundary cases.
  - Kept (per user sign-off): `R` (RefinePlan) — collision with
    `r` (engine cycle) is conceptual not functional since the
    two fire in disjoint operational contexts (active flight vs
    pending transfer). `H` / `I` / `P` / `U` retained.
  - WASDQE attitude + Z / X / Shift+Z / Shift+X throttle stay
    where players (KSP and otherwise) expect.
  - Help overlay + README keybinding tables refreshed.

- **(b) `IterateForTarget` toggle in `m` form.** **Landed v0.8.6.3.**
  Maneuver form grew a 5th cycled field after throttle:
  `iterate (off/on)`. Toggle via space or ←/→ when focused. When
  on, the app routes the commanded Δv through the new
  `World.IterateBurnDV(mode, dv)` sim helper before planting, which
  Newton-iterates against an RK4 finite-burn simulation
  (planner.IterateForTarget). Target picked from mode:
  Prograde/Retrograde → apoapsis, Radial± → periapsis, Normal± →
  no-op (use `I` PlanInclinationChange for plane-rotation Δv).
  Falls back to the commanded Δv on iteration failure so the burn
  always plants. Off by default (impulsive-target semantics fine
  for short / low-loss burns); the value lights up on long / low-
  TWR burns where finite-burn loss is measurable.
- **(c) Multi-rev porkchop UI keys.** **Backlogged.** Defer until
  staging slices grow the craft fleet enough that multi-rev /
  retrograde transfers are practically valuable. The current
  chemical S-IVB-1-class fleet barely reaches Mars on a direct
  nRev=0 prograde Hohmann; multi-rev (nRev≥1) trades transit time
  for lower Δv but is most useful for low-thrust stages (ion
  drives, future content) where the Δv budget is the binding
  constraint. Retrograde transfer is rarely the right pick for
  typical chemical-stage missions. Reopen when introducing
  staging-related craft variants per the state-of-game backlog.
- **(d) Throttle-change warp clamp.** **Landed v0.8.6.2.** Two
  guards added on top of the pre-existing burn-active 10× cap:
  - **Throttle-change cap.** `Spacecraft.LastThrottleChangeAt`
    records sim-time on actual value change in `SetThrottle`;
    `recentlyChangedThrottle(throttleClampWindow=1s)` reports
    "any craft's throttle moved within the last sim-second."
    `clampedWarp` pins to 10× while the window is open.
    Per-craft check (not active-craft-only) so a planted-burn
    throttle ramp on a non-active vessel still triggers.
  - **Upcoming-node approach cap (unplanned add-on).**
    `soonestUpcomingNodeIn` scans every craft's resolved future
    TriggerTime; `clampedWarp` enforces
    `maxWarp = secondsUntilNode / (approachClampSubsteps × BaseStep)`
    with `approachClampSubsteps = 10`, floored at 1×.
    Continuous ramp: 5 s out → 10× (matches burn-active cap so
    the two compose with no discontinuity); 50 s → 100×;
    500 s → 1000×; 5000 s → 10000×. Catches the failure mode
    where 100,000× warp would skip a 30-s-out node in a single
    5000-s tick.

  All three caps + the existing orbital-period cap compose:
  smallest wins.

**Decisions reopen-able later.**

- **(a) Drag-to-edit nodes** stays deferred — click-to-edit-
  replace in v0.6.4 is the current model. Reopen if playtest
  feedback says scrubbing is worth the implementation cost.
- **(e) Attitude-mode save persistence** stays deferred. Today's
  default is "ephemeral, planted nodes are the persistence
  layer". Reopen if players report mid-coast-load resetting
  attitude is annoying.
- **(f) Per-craft keymap chord** folds into v0.8.1 (multi-craft
  foundation handles attitude / throttle keys routing to the
  active craft only).

**Open scoping.**

- Lambert short/long branch flag for multi-rev porkchop.
  Today's solver returns the first root found (lower-z side).
  Pair with the multi-rev UI in v0.8.6, or split into a
  v0.8.6.1 follow-up patch?

**Estimate.** ~250 LOC + tests.
