# v0.7 — modding + manual flight + planner polish

## Context

`docs/state-of-game.md` (snapshot at v0.6.6) frames v0.7 as a single
speculative line ("custom systems + theming") in the §3 version-frame
table. This doc fleshes that out into six concrete slices: a modding
chain (system catalog overlay → per-body color migration → user theme
overrides), a manual-flight-controls strand surfaced during v0.6
playthrough (the v0.6 cycle delivered planner UX but no real-time
stick — the only way to fire the engine today is via a planted
`ManeuverNode`), and the small polish items deferred from v0.6
(inclination-change planner, explicit retrograde flag for
`LambertSolve`).

Multiplayer implementation is explicitly excluded — stays a v0.8+
candidate per the v0.6.6 design spike's "implementation roadmap out
of scope" framing. v0.7.0 / v0.7.1 / v0.7.2 chain because each
consumes the previous schema; v0.7.3 / v0.7.4 / v0.7.5 are
independent and can ship in any order.

## Resolved scoping questions

Settled before writing this plan:

1. **Theme.** Modding (system catalog + palette + theme) + manual
   flight + planner polish. Multiplayer implementation deferred to
   v0.8+ per the v0.6.6 design spike's explicit "implementation
   roadmap out of scope" framing.
2. **Catalog overlay scope (v0.7.0).** User files at
   `$XDG_CONFIG_HOME/terminal-space-program/systems/*.json` merge with
   the embedded set in `bodies.LoadAll`
   (`internal/bodies/systems.go:67`). Conflict policy: user files
   win on `system.Name` match, append otherwise. The existing
   `body_catalog_hash` field at `internal/save/save.go:43` already
   guards against universe drift, so save-rejection on first load
   with a user-supplied system is documented behavior, not a bug.
3. **Color migration (v0.7.1).** Add an optional `Color string` field
   to `bodies.CelestialBody` at `internal/bodies/body.go:20`; fill on
   every embedded body in `internal/bodies/systems/*.json` from the
   existing `bodyPalette` table at `internal/render/palette.go:62`.
   Rewire `render.ColorFor(b)` (`palette.go:98`) to prefer `b.Color`
   when set, fall through to the existing table for backward compat.
   Drop the table next minor (v0.8) once v0.7.1+ saves are in the
   wild.
4. **Theme override granularity (v0.7.2).** One `theme.json` at
   `$XDG_CONFIG_HOME/terminal-space-program/theme.json` with two
   top-level optional blocks: `ui` overlays the UI-tier constants in
   `palette.go`; `bodies` overrides per-body color (wins over
   v0.7.1's per-body `Color` field). Promotion of orbital-element
   overrides to a sibling `bodies.json` is left to v0.8+.
5. **Manual-flight keymap (v0.7.3).** `w/s` prograde/retrograde,
   `a/d` normal±, `q/e` radial±, `z` throttle full / `x` cut,
   `Shift+z` / `Shift+x` ±10 % step. Mode keys select the held
   attitude; press-and-hold any of those keys fires the engine in
   the current attitude; release ends the burn cleanly. Reuses the
   existing `BurnMode` enum (`internal/spacecraft/thrust.go:14`)
   directly — no new enum value.
6. **Save schema for throttle (v0.7.3).** Schema v3 → v4 with
   `Node.Throttle float64,omitempty` (zero unmarshals as 1.0). v3
   saves load with implicit throttle = 1.0 — the existing
   permissive-load contract at `internal/save/save.go:180` already
   accepts older versions.
7. **Inclination planner shape (v0.7.4).** Reuses `BurnNormalPlus`
   / `BurnNormalMinus` (`internal/spacecraft/thrust.go:19-20`); new
   planner picks AN/DN automatically via `orbital.TimeToNodeCrossing`
   (`internal/orbital/events.go:76`, shipped in v0.6.0) and computes
   combined-rotation Δv. No new burn-mode enum value needed.
8. **`LambertSolve` retrograde flag (v0.7.5).** Add a `retrograde
   bool` parameter to `internal/planner/lambert.go:13`; plumb through
   `LambertSolveRev` (line 34) and `planner.PorkchopGrid` so the
   multi-rev porkchop unblock in v0.8+ has both branches. Single-rev
   callers default to prograde (today's behavior).

## Slices

### v0.7.0 — external system catalog overlay

Adds a loader that merges user-supplied JSON systems with the embedded
set, so users can author custom systems without recompiling.

**Code surface.**

- `internal/bodies/systems.go` (`LoadAll`) — extends to also read
  `$XDG_CONFIG_HOME/terminal-space-program/systems/*.json` (or
  `~/.config/terminal-space-program/systems/` if XDG is unset).
- Conflict policy: user files win on `system.Name` match; otherwise
  append. Surfaces a `[]LoadWarning` for any malformed user files
  rather than failing the whole load — embedded systems must always
  load.
- `internal/save/save.go:43` (`BodyCatalogHash`) — no change needed.
  The existing `body_catalog_hash` field already detects catalog
  drift on save load. Documented behavior: a save taken on the
  embedded catalog rejects on first load after a user adds a custom
  system.

**UX.**

- README adds a "Custom systems" section pointing to
  `$XDG_CONFIG_HOME/terminal-space-program/systems/`.
- Body-info screen shows a `source: embedded | user` line so the
  player can tell which catalog a body came from.

**Estimate.** ~150 LOC + tests.

**Shipped.** Tag `v0.7.0`.

- New `LoadAllWithWarnings()` returning `([]System, []LoadWarning,
  error)` is the warning-aware entry point; legacy `LoadAll()` is a
  thin wrapper that drops warnings.
- `System.Source` runtime annotation (`json:"-"` so `CatalogHash`
  stays stable across identical-data overlays). Set to `"embedded"`
  or `"user"` during load.
- `cmd/terminal-space-program/main.go` pre-loads with the
  warnings-aware variant and prints any to stderr before
  `tui.New()` enters bubbletea's alt-screen — otherwise the
  warnings would only become visible after the user quits.
- `internal/tui/screens/bodyinfo.go` sub-header reads
  `<bodyType> in <system> (source: embedded|user)`.
- Four new tests cover empty overlay, append, system-name replace,
  and malformed-file warning. Each isolates via `t.Setenv` +
  `t.TempDir`-based `XDG_CONFIG_HOME` so the developer's real
  overlay dir doesn't leak in.

### v0.7.1 — palette migration to per-body JSON

Promotes the hardcoded `bodyPalette` table to a per-body field on
`CelestialBody`, with backward-compat fallback.

**Code surface.**

- `internal/bodies/body.go:20` (`CelestialBody`) — add optional
  `Color string` field (hex string, e.g. `"#3b82f6"`).
- `internal/bodies/systems/*.json` — fill in colors on every embedded
  body from the values currently in `bodyPalette` at
  `internal/render/palette.go:62`. JSON-only edits; preserves the
  existing palette exactly.
- `internal/render/palette.go:98` (`ColorFor`) — prefer `b.Color`
  when set, fall through to the existing table for backward compat
  (drop the table next minor once v0.7.1+ saves are in the wild).
- Catalog hash changes — saves taken on the v0.6.6 schema reject on
  first load with the existing catalog-mismatch error path. Same UX
  as the v0.5.0-style schema change that introduced moons.

**Why this slice next.** v0.7.0 lands the loader; v0.7.1 fills in
the schema field that lets v0.7.2's overrides target individual
bodies. The chain `loader → field → override` mirrors how v0.6.0's
`TriggerEvent` schema seeded v0.6.5's mission-event hooks.

**Estimate.** ~200 LOC + JSON edits + tests.

**Shipped.** Tag `v0.7.1`.

- `bodies.CelestialBody.Color` is `string` (hex `"#RRGGBB"`),
  `json:"color,omitempty"` — absent on JSON entries that should
  fall through.
- `internal/bodies/systems/sol.json` carries 18 `color` entries
  copied verbatim from the legacy `bodyPalette` table (no visual
  change). Non-Sol systems intentionally left without `color`
  fields; their bodies still fall through to `StellarTint` /
  bodyType-default.
- `render.ColorFor` resolution order: (1) `b.Color` when
  non-empty, (2) `bodyPalette` table fallback, (3) `StellarTint`,
  (4) bodyType default, (5) `ColorTrajectory`.
- `TestColorForKnownBodies` updated to assert `b.Color != ""` for
  every Sol body (was: table entry). New
  `TestColorForJSONFieldMatchesPaletteTable` enforces consistency
  between the two sources of truth during the transition window
  (drop alongside the table in v0.8). New
  `TestColorForPrefersJSONField` +
  `TestColorForFallsBackToTable` cover both branches of the
  resolution order.

### v0.7.2 — user theme overrides

Reads a `theme.json` at startup and overlays the palette tables (UI
tier + per-body) before any rendering.

**Code surface.**

- New `internal/render/theme.go` — reads
  `$XDG_CONFIG_HOME/terminal-space-program/theme.json`. Schema:

  ```json
  {
    "ui":     {"alert": "#ff0000", "warning": "#ffaa00"},
    "bodies": {"earth": "#3b82f6", "mars":    "#dc2626"}
  }
  ```

  Both blocks optional; missing keys fall through to the v0.7.1
  per-body `Color` field, then to the existing `bodyPalette` table.
- `internal/render/palette.go` UI-tier constants — load into
  `palette.LoadUIOverrides(theme.UI)` at startup; lookups go through
  a new accessor instead of the bare constants.
- `internal/render/palette.go:98` (`ColorFor`) — checks `theme.Bodies`
  before falling through to v0.7.1's `b.Color` field.

**Estimate.** ~100 LOC + tests.

**Shipped.** Tag `v0.7.2`.

- New `internal/render/theme.go` with the `Theme` struct (optional
  `ui` + `bodies` blocks) and `LoadTheme()` returning
  `(*Theme, []LoadWarning, error)`.
- UI overrides mutate the existing package-level `Color*` vars in
  place — no call-site changes elsewhere in the codebase. A
  `uiDefaults` map captured at package init lets `LoadTheme` reset
  to defaults on each call so it's idempotent.
- Body overrides land in package-level `bodyOverrides`.
  `ColorFor` resolution order is now (1) theme override, (2)
  `b.Color`, (3) `bodyPalette` table, (4) `StellarTint`, (5)
  bodyType default, (6) `ColorTrajectory`.
- `cmd/main.go` calls `render.LoadTheme()` alongside
  `bodies.LoadAllWithWarnings()` so theme warnings print to stderr
  before bubbletea takes the alt-screen.
- Five tests cover no-file no-op, UI override, body override
  winning over `b.Color`, per-body miss falling through, and the
  idempotent-reset path for repeated `LoadTheme` calls.
- README adds a "Theming" section documenting the file path,
  schema, key naming, and override precedence.

### v0.7.3 — manual flight controls

Real-time stick on top of the planted-node planner. Largest slice in
v0.7; keys off existing RK4 + `BurnMode` paths so the integrator
never sees something new — only **what starts/stops a burn**
changes.

**Throttle.**

- New `Throttle float64` field on `internal/spacecraft/spacecraft.go`
  `Spacecraft` struct (default 1.0).
- Effective thrust = `Thrust * Throttle` in the RK4 burn path
  (`internal/physics/rk4.go`) and in `Spacecraft.BurnTimeForDV`
  (`internal/spacecraft/thrust.go:151`).
- Keys: `z` full / `x` cut / `Shift+z` / `Shift+x` ±10 % step.
- HUD: propellant block adds a one-line throttle readout (`THROTTLE
  100%` ramping to `THROTTLE 0%`).

**Hold-to-burn.**

- New `ManualBurn` field on `internal/sim/world.go` `World`,
  parallel to `ActiveBurn` (which lives at
  `internal/sim/maneuver.go:130`).
- While a hold key is pressed, the integrator routes through the
  same RK4 thrust path used by planted nodes, debiting fuel via the
  same `dm/dt = -Thrust·Throttle/(Isp·g₀)` rule.
- Release ends the burn cleanly (mirrors the `ActiveBurn.EndTime`
  discipline).
- Warp clamps to ≤10× while a manual burn is active — same guard as
  v0.1's planted-burn clamp; reuse the existing helper.

**Attitude hold.**

- Six dedicated keys map directly to existing `BurnMode` constants
  (`internal/spacecraft/thrust.go:17-22`):
  - `w` → `BurnPrograde`
  - `s` → `BurnRetrograde`
  - `a` → `BurnNormalPlus`
  - `d` → `BurnNormalMinus`
  - `q` → `BurnRadialOut`
  - `e` → `BurnRadialIn`
- Hold-to-burn fires along the current `World.AttitudeMode
  spacecraft.BurnMode`. Persists across screens so the HUD can show
  the active hold direction.
- The per-sub-step direction recompute in `internal/physics/rk4.go`
  reuses the existing per-mode logic — prograde-hold tracks the
  rotating velocity frame the same way planted prograde burns
  already do.

**Maneuver-form throttle field.**

- The `m` form gains a fourth cycled field (mode / fire-at / Δv /
  throttle) so planted nodes can carry a per-node throttle override.
- Save schema v3 → v4 with `Node.Throttle float64,omitempty` (zero
  unmarshals as 1.0). v3 saves load with implicit throttle = 1.0.

**Why this slice.** It's the smallest piece of "control the ship
like a game" that's missing — node-driven burns are a planning
surface, not a flight surface. Keying off the existing `BurnMode` +
RK4 thrust paths means the integrator never sees something new;
only the *what starts/stops a burn* changes. Suggested cadence:
**ship this first** in v0.7 since it changes moment-to-moment feel
of the game.

**Estimate.** ~350 LOC + tests.

**Shipped.** Tag `v0.7.3`.

- Hold-to-burn isn't directly implementable in a TUI (no portable
  key-up event) so the model is KSP-style: throttle drives engine
  state, attitude drives direction. Cutting throttle (`x`) ends
  the manual burn in one keypress.
- New `Spacecraft.Throttle float64` field (transient — not
  persisted in the save schema; v0.7.3+ saves implicitly load
  with throttle=1.0). `EffectiveThrottle()` clamps to [0, 1] with
  zero meaning "engine commanded off" — no zero-fallback magic.
- Throttle is plumbed through `MassFlowRate`, `ThrustAccelFn`,
  and `World.stepThrust` so every code path that scales thrust
  honors the player's setting.
- New `ManualBurn` struct on World parallels `ActiveBurn`. Carries
  only `StartTime` (no Δv budget, no end time). Integrator
  dispatches between the two via `thrustingAt` — `ActiveBurn`
  takes priority and StartManualBurn no-ops while one is in
  flight. The warp clamp triggers on either burn type; the
  Kepler-step warp-lock falls back to Verlet for both.
- `World.AttitudeMode` tracks the held attitude. Manual-burn
  integrator path uses `World.AttitudeMode`; planted burns keep
  using `ActiveBurn.Mode`.
- Keymap: `s` (was NextSystem) → tab to free `s` for retrograde;
  `q` (was QuitAsk) → Q to free `q` for radial+. Then ten new
  bindings: `z`/`x` throttle full/cut, `Shift+z`/`Shift+x` ±10 %,
  `w`/`s`/`a`/`d`/`q`/`e` for the six attitude modes. Pressing an
  attitude key with throttle > 0 fires the engine immediately.
- HUD: PROPELLANT block gains a throttle line; new ATTITUDE block
  shows the held mode and manual-burn state ("idle" or "● firing
  T+Xs"). Help overlay gains a MANUAL FLIGHT section.
- Seven new tests cover the lifecycle (start sets ManualBurn;
  no-op during ActiveBurn; no-op at zero throttle; SetThrottle(0)
  stops; AdjustThrottle clamps; warp caps to 10×; fuel-exhaustion
  teardown).

**Deferred (follow-up patch).** Per-node throttle override field
in the maneuver form + save schema bump v3 → v4. Today planted
`ActiveBurn` honors live `Craft.Throttle`, so reducing throttle
mid-flight slows the burn — a scope wart that the per-node field
fixes. Pulled out of v0.7.3 to keep the slice tight and to give
the schema bump its own beat (loading v3 saves with implicit
`Node.Throttle = 1.0` is the kind of thing that wants a focused
PR, not a footnote on a manual-flight feature).

### v0.7.4 — inclination-change planner + HUD pass

Plants a normal± burn at the line of nodes to rotate the orbital
plane, plus a sweep through the right-hand HUD to recover vertical
height now that there are more useful blocks competing for it.

**Code surface (planner).**

- New `internal/planner/inclination.go` — `PlanInclinationChange(
  state, mu, targetIncl, primaryID) (InclinationPlan, error)`.
  Picks the next future AN/DN via `orbital.TimeToNodeCrossing`
  (`internal/orbital/events.go:76`, shipped in v0.6.0). Δv at the
  node = `2 · v_horizontal · sin(|Δi|/2)` with `v_horizontal = h/r`
  — exact for any closed orbit. Returns an `InclinationPlan` with
  `NormalSign ∈ {-1, +1}` so the planner stays free of the
  spacecraft package.
- `internal/sim/maneuver.go` — `World.PlanInclinationChange(
  targetIncl)` adapts to a `ManeuverNode` with mode mapped from
  NormalSign and Duration via `BurnTimeForDV`.
- New `I` keybinding on the orbit screen plants the burn,
  defaulting target to the selected body's inclination (or 0°
  equatorial when none). Mirrors the `H` / `P` "select body, press
  key" UX.
- **Robustness note.** Physical AN/DN identification uses the
  current state's `v.Z` sign rather than the `events.go` AN/DN
  labels. The labels rely on ω, which is degenerate for circular
  orbits where eccentricity-vector cancellation is FP noise. The
  v.Z rule (next equator crossing always flips sign) is correct
  for both circular and elliptical orbits.

**Code surface (HUD pass).**

- VESSEL + PROPELLANT side-by-side at half-width each (with a
  2-char gutter); falls back to stacked rendering when content
  width < 36 cols. ~6 row recovery.
- SYSTEM + SELECTED side-by-side, same shape. ~3 row recovery.
- "view: <mode>" stamped as a `Canvas.SetCellLabel` overlay in the
  bottom-right corner of the orbit canvas (and mirrored on the
  maneuver mini-canvas) — the HUD line under FOCUS is gone. The
  new widget primitive is a plain `(col, row, text)` write into
  `cellOverlays`.
- Inclination line added to PROJECTED ORBIT readouts (orbit HUD +
  maneuver form) — was missing, so AN/DN burns appeared to do
  nothing.
- MISSION block removed from the right-hand HUD; replaced with a
  clickable `[Missions]` button in the orbit-screen title bar
  that opens a dedicated `screens.Missions` list (status glyphs
  ✓/✗/·, name, status text, description).
- `[Menu]` companion button mirrors the existing Esc-on-home
  splash menu. Click cascade: title-bar buttons take priority
  before the existing vessel / node / body / canvas / HUD chain.
- Menu screen reworked with clickable list buttons + Yes/No
  confirm sub-screens for each action; keyboard direct path
  (s / l / q execute immediately) preserved. `[Back]` button on
  menu and missions screens for a click-only return to orbit.

**Code surface (planner-frame fix).**

- `World.HohmannPreviewFor` had been using Sun GM + heliocentric
  distances regardless of target — so a LEO craft's Luna preview
  asked for a Hohmann from ~150M km to ~384k km and quoted Δv1
  ≈ 28 km/s, Δv2 ≈ 242 km/s. Now mirrors `PlanTransfer`'s frame
  selection: intra-primary for moons of craft's primary, "moon →
  parent" note for the reverse, heliocentric default. Same flavour
  as the v0.5.7 fix; never propagated into the preview until now.

**Shipped.** Tag `v0.7.4`. Per-node throttle override + save
schema v3→v4 stays deferred. Mission editor / scripting stays
deferred. Multi-craft selector (referenced by the Missions
screen's single-craft assumptions) stays a v0.8+ concern.

### v0.7.5 — explicit retrograde flag for `LambertSolve`

Today the multi-rev branch in `LambertSolveRev` is selected by

**Code surface.**

- `planner.PorkchopGrid` — add the flag so multi-rev porkchop unblock

- Single-rev callers (`PlanIntraPrimaryHohmann`, the v0.6.4
  click-to-plant path) default to prograde — today's behavior.

**Estimate.** ~80 LOC + tests.

## Sequencing

```
v0.7.0 (overlay loader) ──> v0.7.1 (palette migration) ──> v0.7.2 (theme overrides)

v0.7.3 (manual flight)           ─── independent — any order
v0.7.4 (inclination planner)     ─── independent — any order
```

v0.7.0 / v0.7.1 / v0.7.2 chain because v0.7.1's per-body `Color`
field is the schema addition v0.7.0 locks in, and v0.7.2's body
overrides overlay against it. v0.7.3 / v0.7.4 / v0.7.5 don't depend
on the modding chain or each other; release order is the user's
call.

**Suggested cadence:** ship v0.7.3 (manual flight) early since it
changes the moment-to-moment feel of the game, then the modding
chain, then the remaining polish.

## Deferred to v0.8+

- Multiplayer implementation against the v0.6.6 design.
- Multi-rev porkchop branches (unblocked by v0.7.5).
- Multi-craft control selector — surfaces once a second craft
  exists; the v0.6.6 multiplayer design doc flagged this as a
  sequencing question (open question #1 there).
- N-body perturbations (Lagrangian points, three-body trajectories).
- Multi-system spacecraft (interstellar transfer math or deus-ex
  jump).
- Mission editor / scripting for user-authored objectives — gates
  on the v0.7.0 catalog loader proving out the JSON-overlay UX
  first.
- Optional simple atmospheric drag model for reentry / aerobraking.
- Drag-to-edit on planted maneuver nodes.
- Body-detail rendering (Earth terminator, Mars caps, Jupiter
  banding, Uranus / Neptune ring data).
- Active-burn flame animation.
- HUD typography polish (residual alignment in dense panels).
- Predictor adaptive sampling at high warp (the fixed 96-sample
  horizon collapses to a smear at 10000× on LEO orbits).
- Race-detector CI (gated on `CGO_ENABLED=1` in the build matrix).

## Open questions for the next cycle

Carry-overs from `state-of-game.md` §5 to revisit when v0.7 lands:

1. Whether the v0.7.0 catalog loader should accept
   `bodies.json`-style sibling overlays for orbital-element overrides
   (separate from `theme.json`'s color overlay), or stay
   color-overrides-only at the JSON layer.
2. Whether v0.7.3's `World.AttitudeMode` should persist across
   `Save` / `Load` (today's design says no — attitude is an
   ephemeral stick state, planted nodes are the persistent layer).
3. Whether v0.7.3's hold-to-burn warp clamp should also clamp on
   *throttle change* (today's clamp is on `ActiveBurn != nil ||
   ManualBurn != nil`). A throttle ramp during high warp aliases
   the integrator the same way a held burn does.
4. Theme-file hot-reload — does v0.7.2 watch the file or just load
   at startup? Hot-reload is a v0.8+ polish if the at-startup form
   ships fine.
5. Multi-craft control selector vs multiplayer (carried from v0.6).
   Resolve before committing to a multi-craft v0.8 slice.

Update `docs/state-of-game.md` at each minor / patch boundary so the
snapshot stays current; close out v0.7 cycle questions there once
shipped.
