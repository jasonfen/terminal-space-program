# v0.6 — planner UX + missions + MP design

## Context

`docs/state-of-game.md` (snapshot at v0.5.15) defines v0.6 as four committed
strands: burn-at-next scheduler, mouse support, mission scaffold, and a
multiplayer design-doc spike. Two physics items from §2 backlog fold in:
finite-burn-aware iterative planner, and `PlanTransfer` extension to
non-shared-primary moon transfers.

This doc is the implementation plan. Slices are sequenced so each release
lands a self-contained capability and the orbital-elements work that
v0.6.0 introduces is reused by v0.6.1's HUD readout.

Scope corrections vs. state-of-game.md (the doc framing was stale in two
places):

- **Finite-burn auto-plant.** §2 framed this as the unblocker for v0.5.12's
  27% TLI loss. v0.5.10 actually restored finite burns by upping engine
  power (S-IVB-1, J-2, 1023 kN) and shortening the transfer — the
  underlying long-burn / low-TWR loss is unaddressed. v0.6.2 ships the
  real fix (Newton iteration around the impulsive solver) so future
  low-TWR loadouts work without papering over.
- **Cross-SOI `PlanTransfer`.** §2 said "Earth → Luna and any planet →
  moon." Earth → Luna already works via `PlanIntraPrimaryHohmann`
  (`internal/planner/transfer.go:154`, dispatched at
  `internal/sim/maneuver.go:122`). The actual gap is the **reverse**:
  craft inside a moon's SOI returning to its parent body (Luna → Earth).
  v0.6.3 covers that case.

---

## Resolved scoping questions

Settled before writing this plan:

1. **Trigger model (v0.6.0).** Lazy-freeze. Event-relative nodes resolve
   `TriggerTime` once at the next Tick where the live orbit yields the
   event, then freeze. Matches the existing "trigger time is the moment,
   frozen at plant" model.
2. **Cross-SOI `PlanTransfer` scope (v0.6.3).** Moon → parent only
   (e.g. Luna → Earth). Heliocentric → moon-of-other-planet (Phobos
   from Mars heliocentric, Titan from heliocentric) is not in v0.6.
3. **Finite-burn priority (v0.6.2).** Stays in v0.6. v0.5.10 didn't
   solve the underlying loss; it routed around it for the current
   default vessel.
4. **Mission save shape (v0.6.5).** `Payload.Missions` (inside the
   payload), not a sibling block. Schema bump v1 → v2; v1 saves load
   with empty missions.
5. **Advanced trigger picker (v0.6.0).** No picker. All five event
   modes (`Absolute / NextPeri / NextApo / NextAN / NextDN`) ship as
   one cycle field in the `m` form alongside the existing mode cycle.
6. **Mouse scope (v0.6.4).** Click-only selection. No drag, no
   wheel-zoom. Targets: planet / moon / vessel / maneuver node on
   orbit canvas, plus GUI menu elements (HUD panels, porkchop cells).
7. **Post-burn HUD readout (v0.6.1).** When a maneuver is set, HUD
   shows the predicted post-burn AP/PE/AN/DN. New slice — the
   orbital-elements code path it needs overlaps with v0.6.0's AN/DN
   resolver, so it ships next.
8. **Moon → parent return target altitude (v0.6.3).** Take what you
   get. The escape burn determines the post-drop Earth-frame perigee;
   the user plans circularization manually once in the parent frame.
   A targeted version (Newton-iterated escape Δv for a specified
   parent-frame perigee) can ride v0.6.2's machinery in v0.7+.

## Slices

### v0.6.0 — burn-at-next scheduler

Foundation. Adds event-relative maneuver nodes alongside the existing
absolute-time nodes.

**Code surface.**

- `internal/sim/maneuver.go:34` (`ManeuverNode`) — add
  `Event TriggerEvent` field (default `Absolute` for backwards-compat).
- New `TriggerEvent` enum: `Absolute / NextPeri / NextApo / NextAN /
  NextDN`.
- New resolver in `internal/orbital` (peri/apo come straight from
  elements; AN/DN from `Ω` and the current true longitude). Returns
  the next sim-time the orbit hits the requested event.
- `internal/sim/tick.go` resolves event nodes once at the first Tick
  where the orbit yields a future trigger time, then freezes
  `TriggerTime`. Sort key in `PlanNode` (`internal/sim/maneuver.go:80`)
  uses the resolved time after the first tick — pre-resolution nodes
  sort to the end.
- `internal/save/save.go:80` (`Node`) — add `Event` field; default
  `absolute` on load for v1 saves. Bump `SchemaVersion` to 2.

**UX.**

- `internal/tui/screens/maneuver.go:33` — extend `focus` from 3 to 4
  fields (mode / Δv / duration / fire-at). Fire-at is a 5-way cycle
  matching the existing mode-cycle pattern. When fire-at ≠ Absolute,
  the T+ readout shows "(resolves at first tick)" until the resolver
  fires.
- HUD planned-nodes block tags event-resolved nodes with their event
  name once resolved.

**Estimate.** ~250 LOC + tests.

### v0.6.1 — predicted post-burn orbit HUD

When ≥1 node is planted, HUD shows the post-burn orbit's AP, PE, AN,
DN under the planned-nodes block.

**Code surface.**

- Reuses `internal/sim/predict.go` (already computes the post-burn
  trajectory for the dashed preview) for the post-burn state vector.
- New `internal/orbital` helper that takes (R, V, μ) and returns
  `{Apo, Peri, AscNode, DescNode}`. Same math the v0.6.0 AN/DN
  resolver uses; ship the helper there and call it here.
- HUD render addition in the orbit screen — one new section under
  the planned-nodes list.

**Why this slice next.** AN/DN orbital-elements math is shared with
v0.6.0's resolver, so v0.6.1 is the natural follow-on — the helper
ships in v0.6.0 and gets a second consumer immediately.

**Estimate.** ~150 LOC + tests.

### v0.6.2 — finite-burn-aware iterative planner

Newton-iterating wrapper around the impulsive solver: integrate a
candidate Δv, measure the delivered apoapsis, correct, repeat until
the target is met.

**Code surface.**

- New `internal/planner/finiteburn.go` exposing
  `IterateForTarget(initial impulsive plan, predicate) (refined plan,
  err)`. Predicate is `(state) bool` — generic over "apo at X",
  "peri at Y", or future targets.
- `internal/planner/transfer.go:154` (`PlanIntraPrimaryHohmann`)
  optionally routes through the iterator when called with a
  finite-burn flag. Default off; enabled by callers that need it
  (auto-plant for low-TWR vessels in `internal/sim/maneuver.go`).
- Reuses the existing RK4 burn integrator
  (`internal/physics/rk4.go`).

**Why this stays in v0.6.** v0.5.10's "fix" was to up engine power and
shorten the transfer so the loss became negligible *for the current
vessel*. The underlying long-burn deformation is still there for any
low-TWR loadout (revived ICPS, future ion stages). v0.6.2 closes the
real gap.

**Estimate.** ~250 LOC + tests.

### v0.6.3 — moon → parent escape transfer

New `PlanTransfer` path for the case `target == craft.Primary.ParentID`
(e.g. craft in Luna SOI, target = Earth).

**Geometry.**

- Two-impulse: prograde escape burn at moon-orbit periapsis raising
  apolune past moon SOI, then ballistic drop into the parent frame.
  Parent-frame circularization is *not* planted — the escape Δv
  determines the post-drop perigee; the player circularizes manually
  in Earth frame once the SOI transition fires.
- Phasing: lifted from `PlanIntraPrimaryHohmann` (v0.5.9) — wait for
  the moon's orbital phase that puts the craft on the right side of
  the moon at SOI exit so the parent-frame trajectory drops where
  the player wants.

**Code surface.**

- `internal/sim/maneuver.go` `PlanTransfer` dispatch — new branch
  before the heliocentric-Hohmann fallthrough at line 159, mirroring
  the v0.5.7 dispatch at line 122.
- New `internal/planner/transfer.go` `PlanMoonEscape(...)`. Two-burn
  output: escape burn (in moon frame) + a placeholder arrival node
  (zero Δv, parent-frame, marks the SOI-exit moment so the HUD has a
  trigger to show "you've returned to <parent>").
- Composes with v0.6.0 (escape at next periapsis of moon parking
  orbit) and v0.6.2 (long escape burn benefits from finite-burn
  iteration when TWR is low).

**Estimate.** ~300 LOC + tests.

### v0.6.4 — mouse selection

Click-only input. No drag, no wheel-zoom.

**In scope.**

- Click-to-focus on a body (planet / moon) on the orbit canvas —
  replaces the cycle keystroke as the fast path.
- Click-to-select on the vessel.
- Click-to-select on a planted maneuver node — opens the `m` form
  pre-populated with that node's values.
- Click-to-plant on a porkchop cell (composes with v0.4.1's
  Enter-to-plant — same target action, mouse alternative).
- Click on HUD panels to set focus (e.g. clicking the "selected
  body" block opens the body-info screen).

**Out of scope this release.**

- Drag-to-edit Δv on planted nodes.
- Wheel-zoom on the orbit canvas.
- Drag-to-pan camera.

**Code surface.**

- `internal/tui/app.go` — add `tea.MouseMsg` handler dispatch.
- `internal/tui/widgets/canvas.go` — reverse-projection from cell →
  body / vessel / node. The v0.5.3 per-cell color tagging is the
  natural extension point (each tagged cell already knows what body
  it belongs to; widen the tag to include vessel and node hits).
- Per-screen mouse handlers in `internal/tui/screens/*.go` for
  porkchop and HUD click targets.

**Estimate.** ~150 LOC + tests.

### v0.6.5 — mission scaffold

Pass/fail objectives evaluated against `World` state each Tick.

**Code surface.**

- New `internal/missions/missions.go` — mission state machine,
  `Evaluate(world) MissionStatus` per active mission.
- New `internal/missions/missions.json` — 2–3 starters:
  - "Circularize at 1000 km LEO ±5%" (e ≤ 0.005 over a full revolution).
  - "Luna orbit insertion" (craft.Primary == Luna AND apo within Luna SOI).
  - "Mars SOI flyby" (any tick where craft.Primary == Mars).
- `internal/save/save.go:41` (`Payload`) — add `Missions []Mission`
  field. Schema v1 → v2; v1 saves load with empty missions.
- HUD: one-line status under the active-burn block, showing the
  current mission's name + state (in-progress / passed / failed).

**Estimate.** ~250 LOC + JSON + tests.

### v0.6.6 — multiplayer design-doc spike

Pure writing. `docs/multiplayer-design.md`.

**Sections.**

- **Transport.** WebSocket vs QUIC vs custom UDP; latency / NAT-
  traversal trade-offs.
- **Authority model.** Host-authoritative vs lockstep; implications
  for time-warp coordination (the deal-breaker for shared sandbox —
  warp is a global multiplier, can't trivially desync between
  players).
- **Persistence.** How shared sessions fit the v0.4 save schema.
  Host-authoritative snapshot vs per-player envelope; conflict
  resolution on rejoin; whether a `session` block sits inside or
  alongside `Payload`.
- **Out of scope.** Implementation roadmap, target release.
  Constraints only.

**Estimate.** ~600–1000 words.
