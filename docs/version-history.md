| v0.17.1 | Point release | SOI-entry prediction diagnostics, no save break, no behavior change (PR #127). Playtest-triggered: projected orbits for Hohmann moon transfers wobbled too much to fine-tune against near SOI entry. This release ships the **quantification harness**: a `predictTuning` variant struct over both SOI-aware propagation loops (defaults = shipped behavior) plus an env-gated eval (`TSP_SOI_EVAL=1 go test ./internal/sim -run TestSOIEntryPredictionEval -v`) measuring stability / accuracy / tunability / drawing / cost across Earth→Moon, Kern→Cursor, and Daemon→Byte transfers against the live-flown outcome and a converged reference. Findings: the dashed-line wobble is per-sample-stale body positions at the SOI test (entry-time swing to +3038 s, impact point off 1647 km on fast moons); the capture-preview drift is the node-chain propagator's uncapped period/100 sub-step (bias growing to +6720 s near arrival); hyperbolic-Verlet legs and the live integrator are exonerated. A validated fix candidate (interpolated body positions + bisected crossing + 120 s sub-step cap) zeroes every instability metric at ~1.5× legacy cost — flipping it on is the follow-up slice. |
| v0.17.2 | Point release | The v0.17.1 fix candidate, shipped (ADR 0017, no save break). Both SOI-aware propagators now default to interpolated body positions + a bisection-refined crossing time + a 120 s coast sub-step cap (`defaultPredictTuning`), so the predicted lunar encounter holds still as the craft coasts in: the dashed Projected Orbit's entry-time swing drops 3053→0 s and the capture preview's near-arrival bias +6720→−70 s, giving a stable target to fine-tune a mid-course correction against. The flip raised the dashed-line predictor's worst-case cost to ~10 ms/call, so a **predict-on-change cache** on the orbit screen takes it off the per-frame render path — the node-marker + dashed-leg geometry is memoized and recomputed only when the craft, planted nodes, clock (bucketed), or live orbit change, so paused / low-warp planning is free while a thrust still updates live. Regression tests pin the entry-time stability and the cache behavior; the v0.17.1 `predictTuning` zero value stays as the eval harness's legacy A/B baseline. |
| v0.21.0 | Missions — give the sandbox a purpose | A player-first, data-authored mission system (ADR 0025, PRs #171–#177, save v8→v9). **Model inversion:** an **Objective** is the atomic pass/fail predicate, a **Mission** is an ordered list of Objectives, and a **Program** is a lightweight grouping (a `program` tag + `requires` / `unlocks` edges) — the tutorial and any campaign are both gated Mission chains. **Two objective families:** nine instantaneous *state* predicates (`reach_altitude`, `circularize`, `orbit_insertion`, `circularize_from_pad`, `soi_flyby`, `land_at_body`, `rendezvous`, `dock`, `return_to_body`) plus *event* objectives that match a semantic gameplay action recorded downward from the input layer (`World.RecordAction`, so tutorial steps survive rebinding / layout presets). **Opt-in failure:** `fail_on` (`crashed` / `out_of_fuel`) is the first path that produces `Failed`; declare nothing → retry forever. **`requires` gates evaluation**, not just display, so a later rung can't latch out of order. **Player surface:** the missions screen is a gated ladder — the active mission as a checklist card on top, locked rungs shown with what unlocks them — plus an in-flight **mission checklist chip** on the ADR 0010 chip system (current objective + N/M, a ~4 s `✗ failed: <reason>` flash before advancing, and the step's instruction inline for tutorial rungs), reachable via `M` or the `[Missions]` button. **Content:** an embedded tutorial→challenge ladder — a 3-rung tutorial (orient → plan → fly, taught with event objectives + hint text) and a `requires`-gated challenge ladder (1000 km orbit → Luna flyby → capture → landing & return → dock → Mars flyby). **Opt-in programs:** the Tutorial and Challenge ladder are **off by default** — a fresh sandbox shows no missions, chip, or evaluation until you enable them in `[Menu]` → Settings (persisted to settings.json). **SAVE BREAK (soft):** schema v8→v9 with a re-seed migration — old saves still load, but mission progress resets to the new ladder. |

Cycling focus now zooms a body in far enough that its texture renders instead of the flat placeholder. Earth's look shifts slightly (single-green land, sharper coastlines, no cloud streaks) per the re-baseline. No save break (textures are excluded from the catalog hash). |

| v0.18.0 | Encounter view — SOI Pass prediction + unified markers | **Unified orbital marker glyphs** (ADR 0020, PR #138): every orbital marker is now a single colored glyph stamped at its projected point — ▲ apoapsis (sky cyan), ▼ periapsis (amber), ◇ ascending node (green), ◆ descending node (violet), ⊕ Perilune (amber, red on impact), ✕ closest approach (hot pink), Δ maneuver node (per-leg color). Color encodes type; brightness encodes state (nominal / counterfactual-dim / alarm-red). Retired the chunky `FillDisk` apsis blobs. **Vessel re-glyph** (part of PR #138): every craft collapses to a single `➤` glyph on the orbit map, distinguished by color not shape (old saves keep their stored glyph until re-spawned). **Live SOI Pass forward prediction + Perilune readout** (ADR 0019 A/B/C/E, PR #139): the upcoming SOI encounter of the live, unburned trajectory now renders ahead of arrival — always-on, independent of the Target slot — as a bright foreign-SOI arc with a Perilune ⊕ marker (Impact/red when sub-surface). New always-on **SOI PASS** HUD chip (body, Perilune altitude or IMPACT, Time to Perilune) that de-dupes with the TARGET chip when they name the same body. **Full transit arc** (PR #140): the SOI-pass arc draws the whole transit (entry → perilune → exit), not just the approach. **Dual-arc no-burn counterfactual** (ADR 0019 D, PR #141): with a maneuver node planted, the canvas draws both the bright node-modified path (with its Perilune marker) and a dim no-burn counterfactual capped at the first node — the SOI PASS chip stacks `planned` + `no-burn` rows. **SOI Pass view** (ADR 0019 F, PR #142): the `v` view cycle gains a `soi-pass` mode that frames the active SOI Pass's body and auto-fits its SOI, without setting it as Target. No save break (no schema change; vessel glyph on pre-v0.18 saves updates to `➤` on next re-spawn). |
| v0.18.1 | Point release | Encounter-view framing fix, no save break (#144, PR #145). Playtest-triggered on Kern→Cursor: after burning a combined transfer, focusing the target showed an **empty frame** — the predicted capture orbit wasn't displayed until the maneuvers finished. A framing bug, not a prediction bug: the encounter is drawn at the body's *arrival* position, but every "look at the body" path (`ViewTarget`, `ViewSOIPass`, and plain focus) centered on its *current* position — 269,512 km away (~169× the zoom) for a short-period moon. **Encounter-centered framing**: the three framers now center on the predicted encounter and fit to the **drawn arc's extent**, not the SOI — in the system frame the body translates through the pass, smearing the in-SOI hyperbola across ~24× the SOI, so an SOI-sized fit would frame a single point (verified by rendering). Fitting to the arc extent sidesteps `SOIRadius` (and with it the `Bodies[0]`-root SOI-floor of #143, left as a separate cleanup). Applied to **any view with the orbit visible**, including the plain "focus the body" path. **Planted-transfer reach**: `ViewSOIPass` now sources the planted (node-modified) pass, so the view is reachable and encounter-centered while flying a transfer whose pre-burn orbit can't yet reach the body (where the live pass is false). |
| v0.18.2 | Point release | Manual zoom in the encounter-framed view, no save break (follow-up to #144, PR #146). The v0.18.1 encounter framing re-fit the canvas **every frame**, so `+`/`-` were silently overwritten on the next tick — the view couldn't be zoomed at all. The canvas scale is now composed as `baseScale × userZoom`: auto-fit owns the base, `+`/`-` adjust a multiplier that survives the per-frame refit, and a focus/system change resets the multiplier with the fresh framing context. The landed/primary zoom cap still clamps the result. |
| v0.18.5 | Point release | QWERTZ keyboard-layout preset, no save break (ADR 0022, PR #165 — slice 1 of #130). On a QWERTZ keyboard the physical `Y` and `Z` keys are swapped from QWERTY, so the `z` / `Z` throttle bindings land under the wrong finger. **`Esc → [Controls]`** now offers a keyboard-layout selector (QWERTY ↔ QWERTZ); choosing QWERTZ keeps every binding under the same finger as QWERTY and relabels the F1 help to match your keycaps. Implemented as **ingest-normalize + display-translate** (ADR 0022): a printable keypress is mapped from the active layout back to its QWERTY position at the top of `Update()` before any match — so the keymap and every raw-string screen handler stay authored in QWERTY, untouched — while F1 key tokens (not their descriptions) are translated the other way for display. New leaf `internal/keylayout` package holds a flat injective `y↔z` rune map; the choice persists to `settings.json` via a new `KeyboardLayout` field (no save-schema bump). Letters-only QWERTZ fully fixes the reported binding; AZERTY / Dvorak, punctuation position-mapping, and full per-action remap are deferred (the injective-bijection authoring + validation burden). |
