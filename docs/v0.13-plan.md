> Cycle opened **2026-06-03**, after v0.12 closed (all five grab-bag
> slices shipped; only the out-of-cycle GH #68 lunar-capture tilt
> remains). v0.13 has a **single headline**: kill the orbit-HUD
> overflow. With a target set, a launch in progress, a 7-stage Apollo
> stack, and planted nodes, the HUD renders taller than the canvas and
> the terminal scrolls to the bottom of the bar — hiding the title and
> the orbit view (`orbit.go:846` only ever computed a mouse-offset; it
> never prevented the overflow). The fix, frozen in **ADR 0010**: the
> HUD narrows to a slim always-on column of irreducible vessel
> telemetry, every other block becomes a compact **Chip** composited
> onto a canvas corner like the navball, a new **Settings screen**
> toggles default Chip visibility, and **F2** momentarily declutters.
> Defaults are all-Chips-on, so no information is lost versus today; the
> win is bounded height and no scroll.
>
> **Whole-cycle weight: L** (one feature). It decomposes into four
> dependency-ordered slices: a config layer (foundation), the HUD split
> itself (the headline fix, where the overflow actually dies), the
> Settings screen, and F2 declutter. Sequencing is a chain, not a
> grab-bag — slice 2 is the value; slice 1 is its prerequisite; 3 and 4
> are independent followups once the chip render path exists.

## Slice 1 — Config layer — `Settings` + `settings.json`

<!-- llm-parse: slice=v0.13.x weight=S scope=config status=designed backlog=hud-overlay-chips adr=0010 -->

**Why first.** Both the Settings screen (slice 3) and the Chip render
rule (slice 2) read the same persisted visibility state. Land the data
layer once, with no UI, so the later slices wire into a tested surface
instead of inventing persistence inline.

**Committed scope.**

- New `Settings` struct — chip-visibility booleans (one per Chip:
  Target, Stages, Nodes, Launch, Capture, Projected Orbit, plus the
  navball if it's brought under the same toggle set), with room reserved
  for future non-visibility prefs (units, etc.). Core HUD-column
  telemetry is **not** represented here — it is fixed, never
  configurable (ADR 0010 consequence "Core telemetry is fixed").
- Load/save `settings.json` from `$XDG_CONFIG_HOME/terminal-space-program/`
  (falling back to `~/.config/...`), **mirroring the `theme.json`
  precedent** at `internal/render/theme.go:132` and the bodies overlay
  loader at `internal/bodies/systems.go:193`. Standalone file, not
  folded into `theme.json` (ADR 0010: colour vs. visibility are distinct
  concerns — leaning standalone; see open questions if revisited).
- **Missing file → defaults all-on**, which exactly preserves today's
  behaviour (every block shows). Idempotent loader; unknown/extra JSON
  keys tolerated for forward-compat.
- **Do NOT touch the save schema** (`internal/save/save.go`,
  `SchemaVersion = 6`). UI visibility is an application preference, not
  game state — that's an explicit ADR 0010 rejected alternative
  ("Persist prefs in the save file").

**Test surface (TDD fits well here).** Round-trip a `Settings` through
write→read; a missing file yields the all-on default; a partial/legacy
file fills absent keys with defaults; the loader is idempotent; the
write target resolves under `$XDG_CONFIG_HOME` (and the `~/.config`
fallback when unset). Sibling `_test.go` per the repo convention.

**Where it lives.** New package or file mirroring `internal/render`'s
config pattern — likely `internal/settings` (or alongside the tui if the
struct is UI-shaped). Decide at slice-open against where the consumers
(tui screens) sit; keep the dependency flow one-directional (nothing
lower imports the tui).

**Weight: S.** Pure infra, no rendering risk. Follows two existing
config-loader precedents.

## Slice 2 — HUD split — slim column + canvas Chips

<!-- llm-parse: slice=v0.13.x weight=L scope=tui status=designed backlog=hud-overlay-chips adr=0010 headline=true -->

**The headline.** This is where the overflow bug actually dies. Refactor
`renderHUD` (`internal/tui/screens/orbit.go`, ~`:1157`) into two parts:

1. **The slim always-on HUD column** — core vessel telemetry only: name,
   primary, fuel %, Δv budget, throttle, velocity. Fixed contents,
   bounded width, never declutter-hidden. This is the new, narrowed sense
   of "HUD" (CONTEXT.md §HUD & overlays).
2. **Chip builders** — every other block (`TARGET`, `STAGES`, `NODES`,
   `LAUNCH`, `DESCENT`, `CHUTE`, `CAPTURE PREVIEW`, `PROJECTED ORBIT`,
   `FRAME TRANSITION`, `ATTITUDE`) becomes a compact 2–4 row **Chip**
   composited onto a **Canvas corner**, reusing the navball's compositing
   path (`composeNavballOverlay` ~`orbit.go:813`; panel-dims pattern in
   `navball_panel.go`, but Chips are far smaller than the 34×19 navball).

**Corner map (decided 2026-06-03).** The open chip-to-corner question is
resolved:

| Corner | Occupant |
|---|---|
| **Bottom-left** | **Stages** chip |
| **Bottom-right** | **Navball** (stays put, 34×19), with the **Maneuvers** (Nodes) chip stacked *just above* it in the same right column |
| **Top-right** | **Orbit metrics** chip (the `PROJECTED ORBIT` readout — apo/peri/period/ecc) |
| **Top-left** | free → spillover home for the **phase-transient** chips (`LAUNCH`, `DESCENT`, `CHUTE`, `CAPTURE PREVIEW`, `FRAME TRANSITION`, `TARGET`, `ATTITUDE`), stacked; these are largely mutually exclusive by phase so stacking rarely collides |

The navball **keeps its bottom-right anchor** (`composeNavballOverlay`
`atCol = cCols - navballPanelW`, `atRow = cRows - navballPanelH - 1`); the
Maneuvers chip anchors above it (`atRow - chipH - gap`). Because the
navball is 19 rows tall, "above it" lands roughly mid-right — fine, the
chip is only 2–4 rows. When the navball is declutter-hidden (slice 4) or
absent (no active craft / canvas too small), the Maneuvers chip drops to
the true bottom-right.

**The render rule (load-bearing).** A Chip renders **iff**
`enabled (Settings) && relevant (phase/state) && !declutter`. The
`relevant` gate is the existing phase-driven logic generalised —
`currentHudPhase` / `hudPhaseAscent` / `hudPhaseProximity`
(~`orbit.go:2277`–`:2291`) already drop duplicate rows; the Chip system
supersedes and extends it. In this slice `enabled` reads slice 1's
`Settings` (defaults all-on) and `declutter` is a stubbed-`false`
constant — slice 4 wires the real F2 state.

**Bound the variable-length lists (intrinsic to making them Chips).**

- **Nodes Chip** = next node + `(+N → [m])` count. The full enumerable/
  editable list stays on the **maneuver screen** (`maneuver.go`, `[m]`) —
  that becomes the canonical home (ADR 0010 consequence). Re-point the
  node click-routing that currently depends on HUD layout: `hudNodeHit`
  (~`app.go:889`), `HitHudNode` / `scanHudNodeRows` (`orbit.go`) — node
  clicks now route to the maneuver screen instead of the in-HUD list.
- **Stages Chip** = pips (`●●●○○`) + active N/M. Per-stage detail stays a
  spawn-time concern; the chip summarises.

**Re-widen the canvas.** When the HUD column slims, the canvas should
grow. `OrbitView.Resize` (~`orbit.go:190`–`:197`) currently hard-codes
`canvasCols = totalCols*7/10`. Make the split a function of the slim
column's fixed width rather than a hard 30% — the orbit view reclaims the
space the tall stack used to occupy.

**Delete the dead overflow path.** The `hudScrollOffset` computation
(~`orbit.go:844`–`:857`) only ever fixed mouse coordinates; with bounded
chips there is no overflow to scroll, so it goes (verify the mouse-hit
math no longer needs it before removing).

**Test surface.** The render rule (`enabled && relevant && !declutter`)
unit-tested per Chip; the slim column renders core telemetry at a bounded
height regardless of vessel state; the worst-case scenario (target +
launch + 7-stage Apollo + planted nodes) produces a frame no taller than
the canvas (the regression that motivates the cycle); the Nodes chip
shows `next + (+N → [m])` for N>1; node clicks route to the maneuver
screen; the canvas widens when the column slims.

**Weight: L** — the long pole. `renderHUD` decomposition + a new
multi-chip canvas-compositing layer + corner assignment/stacking + the
canvas re-widen + click-routing changes. **Natural split point if it runs
long:** ship the slim column + a first batch of chips (the always-present
ones — Target, Stages, Nodes) first, add the conditional chips (Capture,
Frame Transition, etc.) as a follow-on tag.

## Slice 3 — Settings screen

<!-- llm-parse: slice=v0.13.x weight=S-M scope=tui status=built backlog=hud-overlay-chips adr=0010 -->

> **Built on `feat/v0.13-config-layer` (2026-06-03).** `SettingsScreen`
> in `internal/tui/screens/settings.go` (cursor nav + space/enter toggle +
> full-row click targets, mirroring `menu.go`'s decoupled-action pattern);
> `screenSettings` enum + View/Update key & mouse dispatch +
> `App.toggleChip` (write-through + immediate `settings.Save`) in
> `app.go`; `MenuActionSettings` + a `[Settings] (t)` entry in `menu.go`
> (no confirm gate — opening a screen is reversible). Tests:
> `screens/settings_test.go` (lists every chip, toggle returns the right
> chip, cursor wrap, click hit-tests) + `app_test.go`
> `TestSettingsScreenRoundTripPersists` (menu→settings→toggle→back round
> trips and persists). Verified in-app: toggling a chip off removes it
> from the canvas and writes `settings.json`. Open questions resolved:
> **persist-on-toggle**; navball left out of `AllChips`.
>
> **Playtest refinement (2026-06-03).** Orbit metrics and the active
> **● BURNS** readout are now **always-on / non-toggleable** — dropped from
> `AllChips` and routed through the empty-id always-on chip path so a
> player can't permanently hide their current orbit or a live burn from
> the Settings screen. They are still **F2-Declutter-hideable** (only the
> pinned VESSEL core survives Declutter — the user picked "neither survives
> F2"). `settings.AllChips` is now 9 toggleable chips. CONTEXT.md §"HUD &
> overlays" updated with the always-on tier.

**Why after slice 2.** A toggle screen that toggles nothing visible is
hollow; once Chips exist and read `Settings.enabled`, the Settings screen
gives the player control over their defaults. (It depends on slice 1 for
the data and slice 2 for the Chips it governs.)

**Committed scope.**

- New `tui/screens/settings.go` sub-model + a `screenSettings` value in
  the screen enum and dispatch in `internal/tui/app.go` (mirror
  `screenOrbit` / `screenMenu` / … wiring).
- A **"Settings" entry in the menu** (`internal/tui/screens/menu.go`,
  alongside the `MenuActionSave` / `Load` / `Quit` pattern).
- The screen lists each Chip with its on/off state; toggling writes
  through to slice 1's `Settings` and **persists** to `settings.json`
  immediately (or on screen-exit — decide at slice-open; immediate is
  simpler and matches "no apply button" expectations).
- Reads from the shared world / config, mutates via the config layer —
  honouring the tui convention that **screens don't mutate world state
  directly** (state changes go through the owning subsystem; here the
  config layer is that subsystem).

**Test surface.** The screen lists every togglable Chip; a toggle flips
the in-memory `Settings` and triggers a persist; re-loading reflects the
toggled state; navigation in/out of the screen (menu → settings → back)
round-trips without losing state.

**Weight: S–M.** A self-contained sub-model + menu entry + dispatch,
following the existing screen-addition precedent. Bounded.

## Slice 4 — F2 Declutter

<!-- llm-parse: slice=v0.13.x weight=XS scope=tui+input status=designed backlog=hud-overlay-chips adr=0010 -->

**Committed scope.**

- One new keybinding — **F2 = Declutter** (the momentary "hide all
  overlays" gesture). `F2` is confirmed free in `internal/tui/input.go`
  (59 bindings already) and matches the KSP "toggle UI" convention
  (ADR 0010 consequence "One new keybinding").
- A transient `declutter bool` on the App (not persisted — Declutter is
  explicitly transient and unsaved, and **never hides the HUD column**;
  CONTEXT.md §Declutter).
- Wire it into the slice-2 render rule: `declutter == true` suppresses
  every Chip (and the navball — see open question on whether the navball
  is included). Flipping the stubbed-`false` constant from slice 2 for
  the real App state.

**Docs (required, per the keybinding source-of-truth convention).**
Update the in-game **`?` help overlay** (the source of truth for
keybindings) **and** `README.md` (which mirrors it) to add F2 =
Declutter. Both, in the same slice.

**Test surface.** With declutter on, no Chip renders and the HUD column
still does; toggling it back restores Chips to their `enabled && relevant`
state; declutter state does not write to `settings.json` (it's transient).

**Weight: XS.** One binding + one transient flag + the render-rule wire +
two doc updates.

## Sequencing

A dependency chain, not a grab-bag — tag in order:

1. **v0.13.0 — Slice 1 (config layer).** Foundation, no UI, no deps.
   Banks the persistence surface the later slices wire into.
2. **v0.13.x — Slice 2 (HUD split).** The headline; where the overflow
   dies. The long pole — has an internal split point (always-present
   chips first, conditional chips second) if it runs long.
3. **v0.13.x — Slice 3 (Settings screen).** After 2, so there are Chips
   to toggle. Independent of slice 4.
4. **v0.13.x — Slice 4 (F2 declutter).** After 2 (needs the chip render
   path). Independent of slice 3 — order 3/4 freely.

**ADR.** ADR 0010 is already written and frozen — it is the contract for
all four slices. No further ADR is expected this cycle unless an open
question (below) forces a decision worth recording.

## Process constraints

- **This is a feature slice → feature branch + PR, not committed to
  `main`.** (Docs-only edits — like this plan — are fine on main; the
  code is not.) Create the branch first.
- CI must stay green: `go vet ./...` then `go test ./... -race -count=1`
  on every push/PR. The repo is test-heavy — add sibling `_test.go`
  alongside each new file.
- The `?` help overlay is the keybinding source of truth and `README.md`
  mirrors it — update **both** when F2 lands (slice 4).
- **Verify in the running app, not just CI.** Build + tmux-drive an
  Apollo stack from the KSC launchpad (the worst-case HUD), confirm the
  frame no longer scrolls and Settings/F2 work. The `/verify` skill
  encodes this recipe.

## Open questions for v0.13

Deliberately left open in ADR 0010 — decide during build; all reversible.

- ~~**Chip-to-corner assignment**~~ (slice 2) — **decided 2026-06-03,
  see the slice-2 corner map.** Stages bottom-left, Maneuvers bottom-right
  stacked above the navball, Orbit metrics top-right, transient chips
  top-left. *Still open within this:* exact stacking/gap when several
  top-left transient chips are simultaneously relevant (rare by phase) —
  decide during build.
- **`settings.json` standalone vs. folded into `theme.json`** (slice 1).
  Leaning standalone (visibility vs. colour are different concerns), but
  revisit if the two configs end up wanting one load path.
- **Is the navball included in F2 Declutter** (slice 4)? CONTEXT.md
  §Declutter currently says Declutter clears "every Chip and the
  Navball"; confirm that's the desired feel in playtest, or keep the
  navball pinned through declutter.
- ~~**Persist-on-toggle vs. persist-on-exit**~~ (slice 3) — **decided
  2026-06-03: persist-on-toggle.** Each space/enter/click toggle writes
  `settings.json` immediately via `App.toggleChip` (write-through to
  `orbitView.Settings()` + `settings.Save`). Simpler than a buffered
  apply, matches the no-apply-button expectation, and a failed write
  flashes the footer while keeping the in-memory edit so the toggle still
  takes effect this session. The navball stayed **out** of `AllChips`
  (it's a Panel, not a Chip) — that incidental question was left as-is,
  not resolved here.

## Deferred / out of scope for v0.13

- **Core-telemetry configurability.** The slim HUD column's contents are
  fixed by design (ADR 0010) — not a future toggle. Out of scope,
  permanently.
- **Per-Chip keybindings.** All visibility control lives in the Settings
  screen; only F2 (hide-all) is a key. Adding per-chip toggle keys to the
  already-crowded 59-key surface is an explicit non-goal.
- **Inclined-transfer wait-vs-Δv median** ([`state-of-game.md` row 12](state-of-game.md#inclined-transfer-wait-vs-Δv-median)) —
  the LEO→Moon ~12-day node-wait knob, reopened 2026-06-03. A separate
  planner thread; **unrelated to the HUD work, don't conflate.**
- **GH #68 lunar-capture tilt** — the out-of-cycle v0.12 carry-over
  (Moon-relative capture geometry); its own future slice, design-first.
